💳 카드 번호 입력 필드 ⌨️ OTP 인증코드 입력 🌀 :user-valid / :user-invalid

💳 카드 번호 입력 필드 — 브랜드 자동 감지·4자리 그룹·만료·CVC 결제·체크아웃 UX

결제 폼의 얼굴인 카드 입력은 그냥 <input> 하나가 아닙니다. 입력하는 동안 4자리마다 띄어쓰기가 붙고, 앞자리로 카드 브랜드(Visa/Master/Amex)를 알아채고, 만료일은 MM/YY로 자동 슬래시가 들어가고, CVC는 3~4자리로 제한됩니다. 아래에 4111…(Visa), 5555…(Master), 3782…(Amex)를 쳐 보세요.

CARD

브랜드는 앞자리로 자동 인식됩니다.

카드 뒷면 서명란의 3자리(Amex는 앞면 4자리).

숫자만 입력해도 자동으로 공백·슬래시가 붙습니다. Amex를 감지하면 그룹이 4-6-5로, CVC 자리수가 4로 바뀝니다.

🎯 이건 뭐고, 요즘은 왜 이렇게 하나

  • 표준 autocomplete 토큰: cc-name / cc-number / cc-exp / cc-csc를 각 칸에 달면 브라우저·비밀번호 관리자가 저장된 카드를 정확히 채워 줍니다. 이름만 잘 붙여도 입력 이탈이 크게 줍니다.
  • 포맷팅은 표시용, 저장은 raw: 화면엔 4111 1111 1111 1111처럼 보이되, 검증·전송 땐 공백을 제거한 순수 숫자로 다룹니다. 커서 튐을 막으려면 값 재설정 후 커서 위치를 보정합니다.
  • 브랜드 감지(BIN 프리픽스): 앞 몇 자리로 카드사를 추정합니다(4=Visa, 51~55/2221~2720=Master, 34·37=Amex 등). 그룹 길이·CVC 자리수·아이콘을 브랜드에 맞춰 바꿔 줍니다. 추정일 뿐 최종 검증은 서버/PG가 합니다.

💻 코드 (복붙용)

// 4자리마다 공백 (Amex면 4-6-5)
function formatCardNumber(raw, brand){
  const d = raw.replace(/\D/g,'').slice(0, brand==='amex'?15:16);
  const groups = brand==='amex' ? [4,6,5] : [4,4,4,4];
  let out='', i=0;
  for(const g of groups){ if(i>=d.length) break;
    out += (out?' ':'') + d.slice(i, i+g); i+=g; }
  return out;
}
// 앞자리로 브랜드 추정 (간이 BIN)
function detectBrand(d){
  if(/^4/.test(d)) return 'visa';
  if(/^(5[1-5]|2[2-7])/.test(d)) return 'mc';
  if(/^3[47]/.test(d)) return 'amex';
  return null;
}
input.addEventListener('input', () => {
  const brand = detectBrand(input.value.replace(/\D/g,''));
  input.value = formatCardNumber(input.value, brand);
});
<!-- 표준 토큰을 붙여야 자동완성이 정확히 채운다 -->
<input autocomplete="cc-number" inputmode="numeric">
<input autocomplete="cc-exp"    inputmode="numeric" placeholder="MM/YY">
<input autocomplete="cc-csc"    inputmode="numeric">

♿ 접근성·상태 설계

  • inputmode="numeric"로 모바일에서 숫자 키패드를 띄웁니다(type="number"는 스피너·앞자리 0 손실 문제로 카드엔 부적합).
  • 브랜드 배지는 색만으로 구분하지 말고 텍스트/아이콘 라벨을 함께 두고, 순수 장식이면 aria-hidden. CVC 도움말은 aria-describedby로 입력과 연결합니다.
  • 이 데모의 유효/무효 테두리는 오늘의 3번 :user-valid를 그대로 씁니다 — 만진 칸만 빨강/초록으로 물듭니다.
paysable 적용: 카드 등록·결제 화면의 핵심 컴포넌트입니다. 브랜드 감지로 그룹·CVC를 맞추고, 표준 cc-* 토큰으로 자동완성을 살리면 체크아웃 이탈률이 내려갑니다. 단 카드 원문 값은 화면 포맷용으로만 두고, 실제 승인은 PG SDK(토큰화)로 넘겨 원문을 서버에 남기지 않는 설계가 안전합니다.

⌨️ OTP 인증코드 입력 — 칸 분리·붙여넣기 분배·자동 이동 입력·폼

SMS·3D Secure 인증에서 흔한 6자리 코드 입력입니다. 칸을 나눠 놓으면 진행 상황이 한눈에 보이고, 한 자 치면 다음 칸으로 자동 이동, 백스페이스로 뒤로, 코드를 통째로 붙여넣으면 칸마다 분배됩니다. 아래에 123456을 붙여넣어 보세요(정답 코드도 123456).

코드가 안 왔나요? 재전송

첫 칸에만 autocomplete="one-time-code"를 붙여 iOS가 SMS 코드를 제안하게 합니다. 6자리가 다 차면 자동으로 확인합니다.

🎯 이건 뭐고, 요즘은 왜 이렇게 하나

  • 왜 칸을 나누나: 진행 상황과 남은 자리수가 즉시 보이고, 각 자리가 커서 오타를 알아채기 쉽습니다. 시각적 리듬 덕에 "몇 자리 코드"라는 인지도 빨라집니다.
  • 붙여넣기 분배가 필수: 사용자는 문자앱에서 코드를 복사해 옵니다. paste 이벤트를 가로채 숫자만 뽑아 칸마다 채우고 마지막 칸으로 포커스를 옮깁니다. 이게 없으면 칸 UI가 오히려 불편해집니다.
  • OS 자동완성: 첫 칸에 autocomplete="one-time-code"를 달면 iOS Safari가 방금 온 SMS의 코드를 키보드 위에 제안합니다(문자에 "code/인증"류 단어 필요). 현재는 Apple 계열에서 주로 동작 — 추정이니 붙여넣기 경로를 항상 함께 지원합니다.

💻 코드 (복붙용)

const boxes = [...root.querySelectorAll('input')];
boxes.forEach((box, i) => {
  box.addEventListener('input', () => {
    box.value = box.value.replace(/\D/g,'').slice(0,1); // 숫자 1자
    box.classList.toggle('filled', !!box.value);
    if(box.value && i < boxes.length-1) boxes[i+1].focus(); // 자동 전진
    checkComplete();
  });
  box.addEventListener('keydown', (e) => {
    if(e.key==='Backspace' && !box.value && i>0){ boxes[i-1].focus(); } // 후진
    if(e.key==='ArrowLeft'  && i>0) boxes[i-1].focus();
    if(e.key==='ArrowRight' && i {           // 붙여넣기 분배
    e.preventDefault();
    const digits = (e.clipboardData.getData('text')||'').replace(/\D/g,'').split('');
    boxes.forEach((b,j)=> b.value = digits[j] || '');
    boxes[Math.min(digits.length, boxes.length)-1].focus();
    checkComplete();
  });
});

♿ 접근성·상태 설계

  • 각 칸에 aria-label("3번째 자리")을 주고, 완료·오류 안내는 role="status" + aria-live="polite"(내용이 바뀌면 스크린리더가 조용히 읽어 주는 영역)로 전달합니다.
  • 포커스 링을 절대 없애지 말 것 — 6칸에선 지금 어디 있는지가 특히 중요합니다. box-shadow 링으로 크게 표시합니다.
  • 모바일 숫자패드(inputmode="numeric"), 화면 낭독기용으로는 "단일 입력 + 시각만 분할" 방식을 쓰기도 합니다(구현 트레이드오프, 추정).
paysable 적용: 로그인 2단계 인증, 결제 3D Secure, 송금 확인 코드에 바로 씁니다. 핵심은 붙여넣기 분배 + 자동 이동 + one-time-code 세 가지 — 이것만 있으면 "칸 나눈 입력"이 번거로움이 아니라 빠른 경험이 됩니다. 재전송 버튼엔 쿨다운 타이머(예: 30초)를 붙여 남용을 막습니다.

🌀 :user-valid / :user-invalid — 사용자가 만진 뒤에만 유효성 색칠 트렌드·개념

예전 :valid / :invalid는 페이지가 뜨자마자 빈 필수 칸을 죄다 빨갛게 칠해 사용자를 다그쳤습니다. 새 :user-valid / :user-invalid는 사용자가 그 칸을 만지고(입력·blur) 난 뒤에만 켜집니다. 아래 두 폼을 비교해 보세요 — 위는 처음부터 빨갛고, 아래는 만진 칸만 물듭니다.

① 옛 방식 :invalid — 건드리지도 않았는데 벌써 빨강(성가심).

📈 왜 지금 뜨나 / 어디에 쓰나

  • "조기 오류" 문제 해결: 폼 검증의 오랜 골칫거리는 너무 일찍 빨갛게 만드는 것이었습니다. :user-valid/:user-invalid는 상호작용 이후에만 매칭돼, JS로 "touched" 상태를 직접 관리하던 코드를 CSS로 대체합니다.
  • Baseline 도달(검색 기준): 2023년 11월부터 Chrome·Edge·Firefox·Safari 전 주요 엔진에서 지원되는 Baseline 기능입니다. 이제 실무 투입 가능.
  • 선언적 검증 UX: input:user-invalid ~ .msg처럼 형제 메시지를 함께 켜고, ::after content:attr(data-err)로 에러 문구까지 CSS로 붙일 수 있습니다(위 데모가 그렇게 동작).

💻 코드 (복붙용)

/* 사용자가 만진 뒤에만 색칠 — JS의 touched 관리 불필요 */
input:user-valid   { border-color: var(--ok);     background: var(--ok-soft); }
input:user-invalid { border-color: var(--danger); background: var(--danger-soft); }

/* 형제 메시지도 함께, 에러 문구는 data-속성에서 */
input:user-valid   ~ .msg::after { content: "✓ 좋아요"; color: var(--ok); }
input:user-invalid ~ .msg::after { content: attr(data-err); color: var(--danger); }
<div class="row">
  <input type="email" required>
  <p class="msg" data-err="올바른 이메일 형식이 아니에요"></p>
</div>
<!-- required·pattern·type 같은 표준 제약만 있으면 브라우저가 판정 -->

♿ 접근성·상태 설계

  • :user-invalid는 시각 표시일 뿐입니다. 스크린리더엔 aria-invalid·에러 텍스트를 aria-describedby로 함께 연결해야 실제로 전달됩니다.
  • 색만으로 구분하지 말고 아이콘·문구를 함께(WCAG 1.4.1). 이 데모는 ✓·텍스트를 같이 씁니다.
  • 구형 브라우저(단종 IE 등)에선 규칙이 무시되니, 기본 상태가 그대로 쓸 만하게 두는 점진적 향상 원칙을 지킵니다.
paysable 적용: 카드 등록·회원가입·정산 계좌 폼에서 "만지기 전까진 조용, 만진 뒤 즉시 피드백" UX를 CSS 몇 줄로 얻습니다. 오늘의 카드 입력 데모(1번)도 이 선택자로 테두리를 칠했습니다. 제출 시점의 전체 요약(어제 다룬 에러 요약)과 조합하면 "칸별 즉시 + 제출 시 종합"이라는 이상적인 2단 검증이 됩니다.
