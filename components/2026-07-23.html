🧭 브레드크럼 — 경로 안내·오버플로 축약·aria-current 내비게이션·탭

브레드크럼(breadcrumb)은 "지금 사이트 어디쯤 와 있는지"를 홈 › 상위 › 현재 형태로 보여 주는 경로 내비게이션입니다. 깊은 대시보드·설정·정산 화면에서 특히 유용하죠. 문제는 경로가 길어지면 한 줄을 넘긴다는 것 — 요즘은 가운데를 …로 접었다가 펼치는 방식으로 풉니다. 아래 버튼으로 깊이를 늘리거나 폭을 좁혀 보세요.

    경로가 4단계를 넘으면 첫 단계와 마지막 2단계만 남기고 가운데를 …로 접습니다. …를 누르면 접힌 단계가 펼쳐집니다. 마지막 항목은 링크가 아니라 aria-current="page"가 붙은 현재 페이지입니다.

    🎯 이건 뭐고, 요즘은 왜 이렇게 하나

    • 구조를 드러내는 보조 내비: 브레드크럼은 주 내비게이션을 대체하지 않습니다. "여기까지 어떻게 왔나 + 한 단계 위로 빠르게" 이동을 돕는 보조 수단입니다. 계층이 깊은(3단계 이상) 사이트에서만 값어치가 있습니다.
    • 오버플로 축약(truncation): 모바일·좁은 폭에서 경로가 넘치면 가운데를 접습니다. 흔한 규칙은 [홈] … [부모] [현재]. 접힌 …는 그냥 텍스트가 아니라 펼치는 버튼이어야 합니다(정보를 숨기기만 하면 안 됨).
    • 마지막은 링크가 아니다: 현재 페이지는 클릭해도 갈 곳이 없으므로 <a>가 아닌 텍스트로 두고 aria-current="page"를 붙입니다. 검색엔진용으로는 BreadcrumbList 구조화 데이터(JSON-LD)를 함께 넣기도 합니다.

    💻 코드 (복붙용)

    <nav aria-label="현재 위치">
  <ol>
    <li><a href="/">홈</a></li>
    <li><a href="/settings">설정</a></li>
    <li><a href="/settings/billing">정산</a></li>
    <!-- 현재 페이지: 링크 아님 + aria-current -->
    <li><span aria-current="page">세금계산서</span></li>
  </ol>
</nav>
    /* 구분자 / 는 CSS로 — DOM 텍스트로 넣지 않기(낭독기가 "슬래시"로 읽음) */
nav.bc li::after{ content:"/"; color:var(--muted); margin:0 6px; }
nav.bc li:last-child::after{ content:""; }
/* 긴 라벨은 말줄임 */
nav.bc a{ max-width:22ch; overflow:hidden; text-overflow:ellipsis; white-space:nowrap; }

    ♿ 접근성·상태 설계

    • 랜드마크 라벨: 감싸는 <nav>에 aria-label="현재 위치"(또는 "breadcrumb")를 붙여 페이지의 다른 nav와 구분되게 합니다. 순서 있는 경로이므로 <ol>을 씁니다.
    • 구분자는 장식: /·›는 ::after content로 넣어 스크린리더가 읽지 않게 합니다. DOM에 문자로 두면 "홈 슬래시 설정"처럼 읽혀 거슬립니다.
    • 접기 버튼: …는 실제 <button>으로, aria-expanded·aria-label="접힌 경로 펼치기"를 주고 키보드로 열 수 있게 합니다.
    paysable 적용: 관리자·정산 콘솔처럼 계층이 깊은 화면(예: 가맹점 › 정산 › 특정 정산건 › 상세)에서 길을 잃지 않게 해 줍니다. 결제 체크아웃 같은 선형 플로우엔 스텝퍼가 맞고, 탐색형 트리 구조엔 브레드크럼이 맞습니다 — 둘을 혼동하지 않는 게 포인트입니다.

    🃏 필터 칩 — 다중 선택·개수 배지·전체 해제로 리스트 좁히기 카드·리스트

    필터 칩(filter chips)은 리스트 위에 얹는 둥근 토글 버튼 묶음입니다. 탭·세그먼트 컨트롤이 "하나만" 고르는 것과 달리, 칩은 여러 개를 동시에 켜서 조건을 조합할 수 있습니다. 아래에서 상태 칩을 눌러 거래 리스트를 실시간으로 좁혀 보세요. 켠 조건은 OR로 합쳐집니다.

    🎯 이건 뭐고, 요즘은 왜 이렇게 하나

    • 칩 vs 탭·세그먼트: 탭/세그먼트는 상호 배타(하나만)이고, 필터 칩은 다중 선택(여러 개 동시에)입니다. "완료 + 대기"를 함께 보고 싶을 때 칩이 맞습니다. 하나만 골라야 하면 세그먼트를 쓰세요.
    • 개수 배지·빈 상태: 각 칩에 해당 개수를 미리 보여 주면 "눌러 봤더니 0건" 헛수고를 줄입니다. 필터 결과가 0건이면 빈 상태(empty state) 문구와 "필터 지우기"를 함께 제공합니다.
    • 전체 해제 = 전체 보기: 아무 칩도 안 켜진 상태를 "전체"로 다룹니다. 별도 "전체" 칩 대신 전체 해제 버튼 하나로 초기화하는 패턴이 요즘 흔합니다.

    💻 코드 (복붙용)

    <!-- 토글이므로 button + aria-pressed (checkbox 대체) -->
<button class="chip" aria-pressed="false" data-f="done">완료</button>

const active = new Set();
chips.addEventListener('click', (e) => {
  const chip = e.target.closest('.chip[data-f]'); if(!chip) return;
  const on = chip.getAttribute('aria-pressed') === 'true';
  chip.setAttribute('aria-pressed', String(!on));
  on ? active.delete(chip.dataset.f) : active.add(chip.dataset.f);
  render();
});
// 켠 게 없으면 전체 통과, 있으면 OR 매칭
const shown = items.filter(it => active.size === 0 || active.has(it.status));

    ♿ 접근성·상태 설계

    • 토글은 aria-pressed: 칩은 "눌린/안 눌린" 두 상태를 가진 토글 버튼입니다. 색(파란 배경)만으로 켜짐을 표현하지 말고 aria-pressed와 ✓ 아이콘을 함께 씁니다(색각·낭독기 대응).
    • 결과 알림: 필터 후 "12건 중 5건 표시"를 role="status" + aria-live="polite" 영역으로 알려 화면을 못 보는 사용자도 결과 변화를 압니다.
    • 키보드·포커스: 칩은 진짜 <button>이라 Tab·Enter·Space가 그대로 동작하고 :focus-visible 링이 보입니다. 칩이 많으면 가로 스크롤 대신 줄바꿈(wrap)을 우선합니다.
    paysable 적용: 거래·정산 내역 화면의 상태(완료/대기/실패)·결제수단·기간 필터에 바로 씁니다. 칩은 URL 쿼리스트링과 묶어 두면(예: ?status=done,pending) 새로고침·공유에도 필터가 유지됩니다. 어제·그제 다룬 레인지 슬라이더(금액)·날짜 프리셋 칩과 조합하면 강력한 복합 필터 바가 됩니다.

    🌀 스크롤 기반 애니메이션 — JS 없이 진행 바·등장 효과 트렌드·개념

    스크롤 기반 애니메이션(scroll-driven animations)은 스크롤 위치 자체를 애니메이션의 "타임라인"으로 삼는 CSS 기능입니다. 예전엔 scroll 이벤트 + JS로 계산하던 걸, 이제 animation-timeline: scroll()·view() 두 줄로 합니다. 아래 회색 상자 안을 스크롤해 보세요 — 위 진행 바가 차오르고, 문단이 뷰에 들어올 때 스르륵 등장합니다.

    👇 이 상자 안을 스크롤하세요. 맨 위 진행 바는 animation-timeline: scroll()로, 아래 카드들은 view()로 뷰포트 진입에 반응합니다.

    1 · scroll() 타임라인스크롤 컨테이너의 진행률(0~100%)을 그대로 애니메이션 진행률로 연결합니다. 읽기 진행 바에 딱.
    2 · view() 타임라인요소가 스크롤포트에 들어오고 나가는 구간을 타임라인으로 씁니다. "보이면 페이드인"을 JS 없이.
    3 · animation-rangeentry 0% entry 55%처럼 "언제부터 언제까지" 애니메이션할지 구간을 지정합니다.
    4 · 메인 스레드 밖스크롤 이벤트 JS와 달리 컴포지터에서 처리돼, 스크롤이 끊기지 않고 부드럽습니다(추정: 브라우저 구현 의존).
    5 · 점진적 향상미지원 브라우저에선 애니메이션만 빠지고 콘텐츠는 그대로 보입니다. 그래서 안전합니다.
    6 · reduced-motionprefers-reduced-motion이면 등장 효과를 끄고 바로 보이게 처리했습니다.

    끝까지 내려오면 진행 바가 꽉 찹니다. 다시 올려 보세요 — 되감깁니다.

    확인 중…

    📈 왜 지금 뜨나 / 어디에 쓰나

    • JS 스크롤 리스너를 대체: 읽기 진행 바, 스크롤 진입 페이드인, 패럴랙스, 가로 스크롤 갤러리 같은 효과를 CSS만으로 구현합니다. scroll 이벤트 핸들러와 IntersectionObserver 상당수를 걷어낼 수 있습니다.
    • 두 종류의 타임라인: scroll()은 스크롤 컨테이너 전체 진행률에, view()는 개별 요소가 화면에 드나드는 구간에 연결합니다. 진행 바는 scroll(), 등장 효과는 view()가 정석입니다.
    • 지원 현황(검색 기준, 2026-07): Chrome·Edge 115+와 Safari 26에서 정식 지원. Firefox는 아직 플래그 뒤(layout.css.scroll-driven-animations.enabled)에 있고 Interop 2026 우선 과제로 올라와 있습니다. 그래서 필수 정보를 애니메이션에 의존시키지 말고 점진적 향상으로 얹는 게 안전합니다.

    💻 코드 (복붙용)

    /* 1) 읽기 진행 바 — 스크롤 컨테이너 진행률에 연결 */
.progress{
  transform-origin:0 50%; transform:scaleX(0);
  animation: grow linear both;
  animation-timeline: scroll(nearest);  /* 가장 가까운 스크롤 조상 */
}
@keyframes grow{ to{ transform:scaleX(1); } }

/* 2) 뷰 진입 시 등장 — 요소가 스크롤포트에 들어오는 구간 */
.reveal{
  opacity:0; transform:translateY(22px);
  animation: in linear both;
  animation-timeline: view();
  animation-range: entry 0% entry 55%;  /* 들어오기 시작~55% 지점 */
}
@keyframes in{ to{ opacity:1; transform:none; } }

/* 3) 모션 최소화 사용자 배려 */
@media (prefers-reduced-motion: reduce){
  .progress{ animation:none; transform:scaleX(1); }
  .reveal{ animation:none; opacity:1; transform:none; }
}

    ♿ 접근성·상태 설계

    • 모션 최소화 존중: prefers-reduced-motion: reduce(전정기관이 예민한 사용자를 위한 OS 설정)를 감지해 등장·이동 효과를 끕니다. 위 데모도 그렇게 처리했습니다.
    • 정보를 애니메이션에 걸지 말 것: 진행 바·페이드인은 장식입니다. 미지원·모션오프 상태에서도 텍스트가 전부 보여야 합니다(점진적 향상). 진행 바엔 aria-hidden.
    • 포커스로도 도달: 스크롤로만 볼 수 있는 콘텐츠는 키보드 사용자도 Tab·화살표로 스크롤 컨테이너에 진입해 읽을 수 있어야 합니다(데모 상자에 tabindex="0").
    paysable 적용: 온보딩·약관·긴 정산 리포트 화면의 읽기 진행 바, 랜딩/공지의 스크롤 등장 효과에 씁니다. 다만 결제·정산처럼 정확성이 생명인 화면에서는 과한 모션을 피하고, 진행 바 정도의 절제된 용도로만 얹는 걸 권합니다. Firefox 미지원을 고려해 항상 애니메이션 없이도 완결되는 마크업을 먼저 만드세요.
