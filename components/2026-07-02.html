🧭 커맨드 팔레트 — ⌘K 로 여는 키보드 우선 내비게이션 내비게이션·탭

커맨드 팔레트(command palette)는 ⌘K(윈도우는 Ctrl K)를 누르면 화면 위에 뜨는 검색창이다. 메뉴를 마우스로 헤집는 대신 이름 몇 글자만 쳐서 페이지 이동·액션 실행으로 바로 점프한다. Linear·Vercel·Raycast가 대중화시킨 이 패턴은 이제 웹앱 내비게이션의 사실상 표준이다. 핵심은 키보드만으로 완결되는 것: 타이핑으로 필터, ↑↓로 이동, Enter로 실행, Esc로 닫기.

⌘K / Ctrl K 를 누르거나 위 칸을 클릭하세요. 열린 뒤 "결제", "환불", "대시" 등을 타이핑해 보세요.

  • 왜 이렇게: 메뉴가 깊어질수록 마우스 내비게이션은 느려진다. 팔레트는 정보 구조 전체를 검색 한 칸으로 평평하게 만들어, 숙련 사용자의 이동 비용을 확 줄인다.
  • 접근성 핵심: 오버레이는 role="dialog" + aria-modal="true". 리스트는 role="listbox", 항목은 role="option", 현재 선택은 aria-activedescendant로 알린다(입력 포커스는 검색창에 유지). 열 때 포커스를 검색창으로, 닫을 때 원래 있던 곳으로 되돌린다.
  • aria-activedescendant = 실제 DOM 포커스는 검색 input에 두면서, "지금 가상으로 선택된 항목이 무엇인지"만 스크린리더에 알리는 속성. 화살표로 목록을 훑어도 타이핑을 이어갈 수 있다.
  • paysable 적용: 관리자/가맹점 콘솔에서 "거래 검색 · 환불 · 정산 리포트 · 설정"으로 점프하는 전역 팔레트. 결제 도메인은 화면 수가 많아 효과가 특히 크다.

🃏 확장형 카드 — 목록은 요약, 누르면 상세가 펼쳐지는 disclosure 카드·리스트

긴 리스트에서 모든 정보를 한 번에 보여주면 눈이 피곤하다. 확장형(disclosure) 카드는 평소엔 핵심 요약만 보이고, 누르면 그 자리에서 상세가 아래로 펼쳐진다. 새 페이지로 넘어가지 않아 맥락을 잃지 않는 게 장점. 요즘은 높이를 픽셀로 계산하지 않고 CSS grid-template-rows: 0fr → 1fr 트릭으로 내용 높이에 맞춰 부드럽게 여닫는다(내용이 몇 줄이든 알아서 맞음).

FAQ·약관·거래 상세처럼 "필요할 때만 펼치는" 곳엔 <details>가 가장 저렴하고 튼튼한 선택입니다.

/* 높이를 픽셀로 안 재고 0fr → 1fr 로 부드럽게 펼치기 */
.txbody      { display:grid; grid-template-rows:0fr; transition:grid-template-rows .24s ease; }
.txcard.open .txbody { grid-template-rows:1fr; }
.txbody-inner { overflow:hidden; }   /* 접혔을 때 내용이 삐져나오지 않게 */
  • 왜 grid 트릭: 예전엔 max-height를 큰 값으로 잡아 어림잡았는데, 내용이 길면 애니메이션이 튀거나 잘렸다. 0fr→1fr은 실제 내용 높이에 정확히 맞춰 여닫혀서 계산이 필요 없다(2026 기준 주요 브라우저 지원).
  • 접근성: 여는 버튼은 진짜 <button>으로, aria-expanded를 true/false로 토글한다. 순정형은 <details>가 이 상태를 알아서 관리한다.
  • prefers-reduced-motion(사용자가 OS에서 "애니메이션 줄이기"를 켠 상태) 이면 전환을 끈다 — 어지럼증 배려.
  • paysable 적용: 거래 내역 리스트에서 한 줄 요약(가맹점·금액·상태)만 두고, 탭하면 승인번호·수수료·정산정보가 그 자리에서 펼쳐지게. 목록 스캔 속도와 상세 접근을 동시에 잡는다.

🌀 View Transitions API — 목록 정렬·필터·보기 전환을 부드럽게 트렌드·개념

리스트를 다시 정렬하거나 필터를 바꾸면 항목들이 툭 하고 순간이동해 어디로 갔는지 눈이 놓친다. View Transitions API(document.startViewTransition())는 DOM을 바꾸는 코드를 감싸기만 하면, 브라우저가 바뀌기 전/후 스냅샷을 떠서 그 사이를 자동으로 애니메이션해 준다. 같은 항목은 새 위치로 스르륵 이동하고, 사라지는 항목은 페이드아웃된다. 예전엔 FLIP 같은 기법을 손으로 짰지만, 이제 업데이트 함수 한 번 감싸는 것으로 끝난다.

⚠️ 이 브라우저는 View Transitions를 지원하지 않아 애니메이션 없이 즉시 갱신됩니다(동작·결과는 동일).

정렬·필터·보기 버튼을 눌러보세요. 남아 있는 항목은 새 위치로 이동, 빠지는 항목은 페이드아웃 됩니다.

/* ① 애니메이션시킬 요소마다 고유 이름을 준다(같은 이름끼리 morph) */
.vt-item { view-transition-name: tx-고유id; }

// ② DOM을 바꾸는 코드를 startViewTransition으로 감싸기만 하면 끝
function update(){
  if (document.startViewTransition) {
    document.startViewTransition(() => render());   // 전/후를 자동 트윈
  } else {
    render();                                        // 미지원 시 즉시 갱신(점진적 축소)
  }
}
  • 왜 지금: 같은 문서(SPA) 내 전환은 2026년 주요 엔진에 모두 안착 — Chrome 111+, Safari 18+, Firefox 144+. 즉 이제 세 엔진 다 지원한다(문서 간 cross-document 전환은 Interop 2026 논의 단계, 추정). (출처: MDN / web.dev / caniuse)
  • view-transition-name = "이 요소를 전환 대상으로 추적해 달라"는 이름표. 같은 이름이 전/후에 모두 있으면 두 위치 사이를 morph, 한쪽에만 있으면 등장/퇴장 애니메이션이 된다. 이름은 화면에 보이는 요소끼리 유일해야 한다.
  • 접근성·성능: 전환 중 스냅샷은 렌더만 바꿀 뿐 DOM 구조·포커스는 그대로다. prefers-reduced-motion(OS의 "동작 줄이기")이 켜져 있으면 ::view-transition-group의 애니메이션을 꺼서 즉시 전환한다.
  • 점진적 축소: document.startViewTransition이 없으면 그냥 render()를 바로 호출 — 애니메이션만 빠지고 기능은 동일. "안 되면 깨지는" 위험이 없다.
  • paysable 적용: 거래 내역의 정렬·기간 필터 전환, 리스트↔카드 보기 토글, 대시보드 위젯 재배치. 특히 금액순↔최신순처럼 순서가 크게 바뀌는 화면에서 사용자가 항목을 눈으로 따라갈 수 있게 해준다.
