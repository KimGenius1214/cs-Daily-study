🔘 아이콘 버튼 · 툴팁 버튼·액션

공간이 좁은 툴바에서 자주 쓰는 아이콘 전용 버튼. 예뻐 보이지만 접근성 지뢰밭입니다 — 라벨·툴팁·토글 상태를 어떻게 처리하느냐가 4년차의 실력 차이를 만듭니다.

좋아요 128 · 북마크 안 함

각 버튼에 마우스를 올리거나 키보드로 포커스(Tab) 하면 툴팁이 뜹니다. 하트/북마크는 눌러서 상태가 토글됩니다.

왜 이렇게 하나

  • 아이콘만 있는 버튼엔 반드시 접근 가능한 이름을 준다. 여기선 aria-label. 이게 없으면 스크린리더 사용자에겐 "버튼"으로만 읽힌다.
  • 툴팁은 보조 수단일 뿐, 유일한 라벨이면 안 된다. 그래서 툴팁 텍스트에는 aria-hidden="true"를 걸어 aria-label과 중복 읽힘을 막는다.
  • 툴팁은 hover뿐 아니라 :focus-visible에서도 떠야 한다(키보드 사용자). 마우스 없이도 라벨을 볼 수 있어야 한다.
  • 토글 버튼(좋아요·북마크)은 aria-pressed로 "켜짐/꺼짐"을 노출한다. 색만 바꾸면 스크린리더는 상태 변화를 모른다.

💻 코드 (복붙용)

<!-- 액션 아이콘 버튼: aria-label 필수, 툴팁은 aria-hidden -->
<button class="iconbtn" aria-label="삭제">
  🗑️ <span class="tip" aria-hidden="true">삭제</span>
</button>

<!-- 토글 버튼: aria-pressed 로 상태 노출 -->
<button class="iconbtn" id="likeBtn" aria-pressed="false" aria-label="좋아요">🤍</button>
<script>
  likeBtn.addEventListener('click', () => {
    const on = likeBtn.getAttribute('aria-pressed') === 'true';
    likeBtn.setAttribute('aria-pressed', String(!on));
    likeBtn.firstChild.textContent = on ? '🤍' : '❤️';  // 시각 상태
  });
</script>
/* 툴팁: hover + focus-visible 둘 다에서 표시 */
.iconbtn .tip{opacity:0;pointer-events:none;transition:opacity .15s}
.iconbtn:hover .tip,
.iconbtn:focus-visible .tip{opacity:1}
@media (prefers-reduced-motion:reduce){ .iconbtn .tip{transition:none} }

♿ 접근성·상태 설계

  • 툴바는 role="toolbar" + aria-label로 묶는다(정교하게 하려면 방향키 이동까지).
  • WCAG 1.4.13 — 호버/포커스로 뜬 툴팁은 사라지지 않고 유지되고 Esc로 닫을 수 있어야 한다(추정: 브라우저/구현별 차이 있어 실제 검증 권장).
  • 토글 상태는 색 + aria-pressed + (가능하면) 아이콘 모양 변화를 함께 — 색맹 사용자 배려.
paysable 적용: 거래 내역 행의 편집·영수증·삭제 아이콘, 카드 즐겨찾기 토글. 결제 화면은 공간이 빡빡해 아이콘 버튼을 남발하기 쉬운데, aria-label 없는 아이콘 버튼은 접근성 감사에서 바로 걸린다. 툴팁만 믿지 말 것.

⌨️ 레인지 슬라이더 — 금액·범위 필터 입력·폼

"대략 이 정도"를 고르게 하는 입력. 채워지는 트랙과 실시간 값 표시로 지금 얼마인지 즉시 보여줍니다. 단일 슬라이더와 두 손잡이(최소~최대) 범위 슬라이더를 실제로 만져보세요.

일시불

0 · 3 · 6 · 9 · 12개월 단계. 채워지는 트랙은 ::-moz-range-progress(파이어폭스)와 JS 그라데이션(크롬)으로 처리 — 브라우저마다 방식이 다릅니다.

결제 금액 범위 10,000 ~ 500,000원

두 손잡이가 서로 교차하지 않게 막습니다. 칩을 누르면 범위가 그 값으로 점프합니다.

왜 이렇게 하나 / 변형 판단

  • 단일 슬라이더: 정확한 숫자보다 "정도"가 중요할 때(밝기·개월·비율). 정확한 금액 입력이면 숫자 입력창이 낫다 — 슬라이더는 미세 조정이 어렵다.
  • 두 손잡이 범위: "얼마~얼마" 필터(가격·기간). 네이티브 input[type=range] 두 개를 겹쳐서 만들면 키보드·스크린리더 지원이 공짜로 따라온다(라이브러리 없이).
  • 겹침 방지: min 손잡이가 max를 넘지 못하게 값 보정. 채움(fill) 바는 두 값 사이만 칠한다.

💻 코드 (두 손잡이 겹침 방지 + 채움)

const min = document.getElementById('rMin');
const max = document.getElementById('rMax');
const fill = document.getElementById('dualFill');
const GAP = 10000; // 최소 간격

function update(){
  let lo = +min.value, hi = +max.value;
  if (lo > hi - GAP){                 // 손잡이 교차 방지
    if (document.activeElement === min) min.value = hi - GAP;
    else max.value = lo + GAP;
    lo = +min.value; hi = +max.value;
  }
  const p = 100 / (+max.max);          // 0~max 를 % 로
  fill.style.left  = lo * p + '%';
  fill.style.width = (hi - lo) * p + '%';
  label.textContent = fmt(lo) + ' ~ ' + fmt(hi) + '원';
}
min.addEventListener('input', update);
max.addEventListener('input', update);

♿ 접근성·상태 설계

  • 네이티브 range는 방향키·Home/End·스크린리더 값 읽기를 기본 제공한다. 두 손잡이엔 각각 aria-label("최소 금액"/"최대 금액").
  • 현재 값은 눈에 보이는 텍스트(aria-describedby 연결)로도 항상 노출 — 슬라이더만 보고는 정확한 값을 모른다.
  • 손잡이 :focus-visible 링, 충분한 크기(터치 44px 권장), 채움 대비 확보.
paysable 적용: 거래 검색의 금액 범위 필터, 할부 개월 선택, 한도 설정. 정확한 값이 필요하면 슬라이더 + 숫자 입력을 양방향으로 묶어 둘 다 제공하는 게 결제 도메인에선 안전하다.

🌀 Invoker Commands — JS 없이 버튼으로 팝오버·다이얼로그 열기 트렌드·개념

Invoker Commands(호출자 커맨드)는 버튼에 command·commandfor 두 속성만 달면, 이벤트 리스너 한 줄 없이 팝오버·다이얼로그를 열고 닫는 HTML 기능입니다. 2026년 Baseline(주요 브라우저 공통 지원)에 도달했습니다.

결제를 취소할까요?

이 작업은 되돌릴 수 없습니다. 승인된 금액은 3~5영업일 내 환불됩니다.

위 "더보기 메뉴"·"결제 취소" 버튼은 이 페이지의 JS와 무관하게 순수 HTML 속성만으로 동작합니다. (지원 안 되는 옛 브라우저라면 아래 폴백 안내가 뜹니다.)

⚠️ 이 브라우저는 Invoker Commands를 아직 지원하지 않습니다 — 실제 서비스라면 JS 폴백이 필요합니다.

📈 왜 지금 뜨나 / 어디에 쓰나

  • JS 없이 상호작용 — 스크립트 다운로드·실행을 기다리지 않아 초기 반응성이 좋다. 접근성(포커스 이동, Esc 닫기, top-layer 배치)도 브라우저가 알아서 처리한다.
  • Baseline 2025 진입 — Chrome 135, Firefox 144, 그리고 Safari 26.2로 주요 브라우저 지원이 완성됐다(검색 기준). 이제 프로덕션에서 점진적 도입을 고려할 단계.
  • 팝오버용 커맨드: toggle-popover, show-popover, hide-popover. 다이얼로그용: show-modal, close, request-close.
  • --로 시작하는 커스텀 커맨드도 정의할 수 있다(선언적으로 배선하고 동작만 JS로).

💻 코드 (복붙용) — 이게 전부입니다

<!-- 팝오버: 토글 -->
<button commandfor="menu" command="toggle-popover">더보기</button>
<div id="menu" popover> … </div>

<!-- 다이얼로그: 모달로 열기 / 닫기 -->
<button commandfor="dlg" command="show-modal">결제 취소</button>
<dialog id="dlg">
  <button commandfor="dlg" command="close">닫기</button>
</dialog>

<!-- 커스텀 커맨드는 JS로 수신 -->
<button commandfor="cart" command="--add-item">담기</button>
<script>
  cart.addEventListener('command', (e) => {
    if (e.command === '--add-item') { /* ... */ }
  });
</script>

♿ 접근성·주의

  • 브라우저가 aria-expanded·포커스 관리를 상당 부분 대신 해주지만, 메뉴 항목엔 여전히 role="menuitem" 등 시맨틱을 챙긴다.
  • 결제·금융처럼 구형 브라우저 비중이 있는 환경이면 기능 감지 후 폴백이 필요하다(위 데모의 폴백 문구가 그 예 — 추정: 실제 프로덕션은 사용자층에 맞춰 판단).
paysable 적용: "결제 취소 확인 다이얼로그", 거래 행의 "⋯ 더보기 메뉴" 같은 곳부터 선언적으로 얹어 JS 표면적을 줄일 수 있다. 다만 파괴적 동작은 request-close로 "정말 닫을까 / 제출 중 보호"를 함께 설계한다.
