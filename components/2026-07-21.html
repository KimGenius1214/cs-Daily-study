🪟 슬라이드오버 패널 — 목록에서 상세를 옆에서 열기 오버레이·모달·토스트

화면 가운데를 덮는 모달 대신, 오른쪽에서 스윽 밀려 들어오는 패널로 상세를 보여주는 패턴입니다(슬라이드오버 / 사이드 시트). 목록 맥락을 유지한 채 상세를 확인·편집하기 좋아 대시보드·핀테크에서 표준이 됐습니다. 거래 한 줄을 눌러보세요.

행을 누르면 오른쪽에서 상세 패널이 열립니다. Esc 또는 바깥 클릭, ✕ 버튼으로 닫히고, 닫으면 눌렀던 행으로 포커스가 되돌아갑니다.

🎯 이건 뭐고, 요즘은 왜 이렇게 하나

  • 모달과 뭐가 다른가: 가운데 모달은 화면을 완전히 가로막아 "이것부터 처리"를 강제합니다. 슬라이드오버는 옆에서 열려 목록 맥락을 남기며, 항목을 이어 훑어보기(다음 거래 클릭) 좋습니다.
  • native <dialog> 재활용: 모달용 <dialog>를 그대로 쓰되 CSS로 오른쪽에 붙이고 슬라이드시킵니다. showModal()이 포커스 트랩(초점을 패널 안에 가둠), Esc 닫기, 배경 ::backdrop, 배경 비활성화를 공짜로 처리합니다.
  • 포커스 복귀: 닫을 때 열었던 그 행으로 포커스를 되돌립니다. 키보드 사용자가 "어디였지" 하고 헤매지 않게 하는, 눈에 안 띄지만 중요한 디테일입니다.

💻 코드 (복붙용)

<dialog class="sheet" id="sheet">
  <div class="sheet-inner"> … 상세 내용 … </div>
</dialog>
/* dialog 는 전체를 덮고, 실제 패널만 오른쪽에서 슬라이드 */
dialog.sheet{width:100%;height:100%;background:transparent;border:none}
dialog.sheet::backdrop{background:rgba(10,12,16,.45)}
.sheet-inner{position:fixed;inset:0 0 0 auto;width:min(400px,90vw);
  transform:translateX(100%);transition:transform .28s}
dialog.sheet[open] .sheet-inner{transform:translateX(0)}
@media (prefers-reduced-motion:reduce){.sheet-inner{transition:none}}
let opener = null;
row.addEventListener('click', () => {
  opener = row;              // 포커스 복귀용으로 기억
  fillDetail(row.dataset);
  sheet.showModal();         // 포커스 트랩 + Esc + backdrop 자동
});
sheet.addEventListener('close', () => opener && opener.focus()); // 복귀
// 바깥(backdrop) 클릭으로 닫기
sheet.addEventListener('click', e => {
  if (e.target === sheet) sheet.close();   // inner 바깥 = backdrop
});

♿ 접근성·상태 설계

  • showModal()로 열면 aria-modal 의미가 적용돼 배경이 보조기기에서 비활성화됩니다. 패널엔 aria-labelledby로 제목 연결.
  • 포커스 복귀(위 코드)와 Esc 닫기는 필수. 닫힘 애니메이션 때문에 close()를 미루면 포커스 타이밍이 꼬일 수 있어 주의(추정: 구현별 검증 권장).
  • 모션 최소화(prefers-reduced-motion: 사용자가 OS에서 애니메이션 줄이기를 켠 상태) 시 슬라이드를 끕니다.
paysable 적용: 거래 내역에서 행 클릭 → 상세/영수증을 옆 패널로. 목록을 떠나지 않아 여러 건을 빠르게 검토할 수 있습니다. 편집 폼을 패널에 담을 땐 "저장 안 함 상태로 닫기"에 확인을 붙이고, 모바일에선 같은 <dialog>를 바텀시트로 바꿔 반응형 처리하면 됩니다.

🔔 폼 에러 요약 — 제출 실패를 한 번에 모아 보여주기 피드백·상태(로딩·빈상태·에러)

긴 결제 폼에서 제출을 눌렀는데 여러 칸이 틀렸을 때, 각 칸 밑에 에러를 흩뿌리기만 하면 사용자는 뭐가 몇 개 틀렸는지 못 봅니다. 에러 요약(error summary) 배너를 폼 맨 위에 띄우고 거기로 포커스를 옮기는 게 접근성 모범 패턴입니다. 빈칸으로 제출해 보세요.

입력을 확인해 주세요

    이름을 입력해 주세요.
    카드 번호 16자리를 정확히 입력해 주세요.
    유효기간을 MM/YY 형식으로 입력해 주세요.

    ✅ 통과! 실제라면 여기서 결제 승인을 진행합니다.

    제출하면 상단 요약 배너로 포커스가 이동합니다. 배너의 각 항목을 클릭하면 해당 입력 칸으로 바로 점프합니다.

    🎯 이건 뭐고, 요즘은 왜 이렇게 하나

    • 흩뿌린 인라인 에러의 한계: 칸 밑 메시지는 그 칸을 볼 때만 유용합니다. 화면 밖으로 스크롤됐거나 개수가 많으면 사용자는 전체 그림을 못 봅니다.
    • 에러 요약 배너: 폼 맨 위에 "3개 항목을 확인하세요" 처럼 모아서 보여주고, 각 항목은 문제 입력으로 가는 링크입니다. GOV.UK·많은 디자인 시스템의 표준 패턴입니다.
    • 포커스 이동이 핵심: 제출 실패 시 요약 배너(tabindex="-1": 클릭 순서엔 안 들어가지만 스크립트로 포커스는 줄 수 있게)에 focus()를 줍니다. 스크린리더 사용자는 즉시 "오류가 몇 개 있다"를 듣고, 시각 사용자는 배너가 눈에 들어옵니다.

    💻 코드 (복붙용)

    function onSubmit(e){
  e.preventDefault();
  const errors = validate();           // [{id:'cardNo', msg:'…'}, …]

  // 각 칸 표시
  fields.forEach(f => f.setAttribute('aria-invalid',
     errors.some(er => er.id === f.id)));

  if (errors.length){
    // 요약 배너 채우고 → 포커스 이동 (스크린리더가 즉시 읽음)
    renderSummary(errors);             // <a href="#cardNo">…</a> 목록
    summary.classList.add('show');
    summary.focus();                   // tabindex="-1" 이라 포커스 가능
    return;
  }
  submitPayment();
}

// 요약 링크 클릭 → 해당 입력으로 포커스
list.addEventListener('click', e => {
  const a = e.target.closest('a'); if(!a) return;
  e.preventDefault();
  document.querySelector(a.getAttribute('href')).focus();
});

    ♿ 접근성·상태 설계

    • 배너는 role="alert"(또는 aria-live: 내용이 바뀌면 스크린리더가 자동으로 읽는 영역)로 즉시 알림. 문제 입력엔 aria-invalid="true" + 에러 메시지를 aria-describedby로 연결(이 데모는 인접 요소로 간단화).
    • 색만으로 오류를 표시하지 않기 — 아이콘·텍스트·테두리를 함께(색각 이상 배려, WCAG 1.4.1).
    • 제출 시점에 요약을, 개별 칸은 사용자가 만진 뒤(blur/입력)에 검증하는 게 덜 성가십니다.
    paysable 적용: 카드 등록·결제 폼, 정산 계좌 등록처럼 칸이 많고 규칙이 빡빡한 폼에 그대로 맞습니다. 서버 검증 실패(예: 카드 거절)도 같은 요약 배너에 얹어 클라이언트·서버 오류를 한 UI로 통일하면 사용자가 헷갈리지 않습니다.

    🌀 :has() — 부모·형제를 선택하는 관계형 선택자 트렌드·개념

    :has()는 "안에 ○○를 가진 요소"를 고르는 CSS 선택자입니다. 오랫동안 없던 "부모 선택자"가 드디어 생긴 셈이라, 예전엔 JS로 클래스를 토글하던 상태 기반 스타일을 CSS만으로 처리합니다. 아래 둘 다 JS 한 줄 없이 동작합니다.

    ① 선택된 카드 하이라이트 — 라디오를 고르면 그 카드가 켜집니다(.plan:has(input:checked)).

    ② 폼이 유효할 때만 버튼 활성 — 컨테이너가 "안에 :invalid 입력이 있는지"를 :has()로 보고 버튼을 잠급니다.

    ✅ 적용할 수 있어요

    확인 중…

    📈 왜 지금 뜨나 / 어디에 쓰나

    • 부모/형제 선택: .card:has(img) = "이미지를 가진 카드", label:has(input:checked) = "체크된 입력을 가진 라벨". 자식·상태를 보고 조상을 꾸밀 수 있어, JS로 클래스를 붙였다 떼던 일이 사라집니다.
    • Baseline 도달(검색 기준): 2023년 말부터 Chrome 105+, Firefox 121+, Safari 15.4+에서 지원돼 이제 어디서나 쓸 수 있습니다. IE는 미지원(단종).
    • 상태 기반 스타일이 선언적으로: 폼 유효성(:has(input:invalid)), 선택 상태, "빈 목록일 때(:has(:empty)) 안내 문구" 같은 UI를 CSS 안에서 끝냅니다 — 버그날 자바스크립트가 줄어듭니다.

    💻 코드 (복붙용)

    /* ① 체크된 라디오를 가진 카드(부모)를 하이라이트 */
.plan:has(input:checked){
  border-color: var(--brand);
  background: var(--brand-soft);
}

/* ② 폼 안에 잘못된 입력이 하나라도 있으면 제출 버튼 잠금 */
.mini-form:has(input:invalid) .go{
  opacity:.5; pointer-events:none;
}

/* 응용: 이미지 있는 카드 vs 없는 카드 레이아웃 분기 */
.card:has(img){ grid-template-columns: 120px 1fr; }

    ♿ 접근성·상태 설계

    • :has()는 보이는 스타일만 바꿉니다. 버튼을 흐리게만 하지 말고 실제로도 못 누르게(위 pointer-events:none) 하거나, 진짜 잠금이 필요하면 disabled/aria-disabled를 함께 둡니다 — 시각 상태와 실제 동작을 일치시킵니다.
    • 선택 상태는 색만이 아니라 테두리·체크 표시 등 형태로도 구분(색각 이상 배려).
    • 미지원 브라우저(구형)에선 :has() 규칙이 통째로 무시되니, 기본 상태가 그 자체로 쓸 만하게(점진적 향상) 설계합니다.
    paysable 적용: 결제수단·요금제 선택 카드 하이라이트, 카드 입력 폼의 "모두 유효할 때만 결제 버튼 활성", "거래 없음일 때 빈 상태 문구 노출" 등에 바로 씁니다. 상태 관리 코드를 CSS로 밀어내 컴포넌트가 가벼워집니다. 단, 진짜 잠금·검증은 JS/서버가 최종 책임 — CSS는 표현일 뿐입니다.
