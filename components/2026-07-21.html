🪟 팝오버 · 앵커 포지셔닝 — 트리거에 딱 붙는 메뉴·툴팁 오버레이·모달·토스트

드롭다운 메뉴·툴팁을 버튼 바로 옆에 붙이고, 화면 밖으로 넘칠 것 같으면 반대편으로 자동으로 뒤집는 일 — 예전엔 Floating UI 같은 JS 라이브러리가 필요했습니다. 이제 Popover API + CSS 앵커 포지셔닝(anchor positioning)으로 JS 없이 됩니다. 세 트리거를 눌러보세요.
가맹점 표기가 다른가요? 카드사 명세서엔 대표 가맹점명으로 찍힐 수 있어요. 실제 결제처와 금액은 영수증 내역에서 확인하세요.
이 팝오버는 트리거 위에 붙습니다. 위 공간이 부족하면 아래로 flip 됩니다.

각 버튼은 popovertarget 속성 하나로 열리고(바깥 클릭·Esc로 닫힘), 위치는 CSS position-anchor가 잡습니다. 스크롤해도 트리거를 따라다닙니다.

확인 중…

🎯 이건 뭐고, 요즘은 왜 이렇게 하나

  • Popover API: popover 속성을 단 요소는 브라우저 top layer(항상 맨 위 레이어)에 뜨고, 바깥 클릭으로 닫힘(light-dismiss)·Esc 닫힘·포커스 관리를 기본 제공합니다. 여는 버튼은 popovertarget 하나면 끝(2024년 Baseline·주요 브라우저 공통 지원).
  • 앵커 포지셔닝(anchor positioning): "이 팝오버를 저 버튼 기준으로 배치" 를 CSS만으로 선언합니다. 버튼에 anchor-name, 팝오버에 position-anchor + anchor() 값. 스크롤·리사이즈에도 자동으로 따라붙어, 좌표를 JS로 계산하던 코드가 사라집니다.
  • 자동 flip: position-try-fallbacks: flip-block — 팝오버가 화면 밖으로 넘칠 것 같으면 위/아래(또는 좌/우)를 알아서 뒤집습니다. 툴팁 라이브러리의 핵심 기능이 CSS 한 줄로.

💻 코드 (복붙용)

<!-- 여는 버튼: popovertarget 하나면 끝 -->
<button id="trigger" popovertarget="menu">계정 메뉴</button>
<div id="menu" popover> … </div>
/* 버튼을 앵커로 등록 */
#trigger { anchor-name: --menu-anchor; }

/* 팝오버를 그 앵커의 '아래·가운데'에 배치, 넘치면 flip */
#menu {
  position: fixed;
  position-anchor: --menu-anchor;
  top: anchor(bottom);          /* 앵커의 아래 모서리에 top 정렬 */
  justify-self: anchor-center;  /* 가로 가운데 */
  margin-top: 8px;
  position-try-fallbacks: flip-block;  /* 넘치면 위로 뒤집기 */
}

♿ 접근성·상태 설계

  • Popover는 키보드·포커스·Esc 닫기를 브라우저가 처리하지만, 메뉴라면 role="menu"/role="menuitem" 시맨틱은 직접 챙깁니다.
  • 지원 범위(검색 기준): 앵커 포지셔닝은 Baseline 2026으로 Chrome 125+, Firefox 132+, Safari 18.2+ 지원. 단 flip 폴백(position-try)은 Safari 18.4+부터 완전합니다. 그래서 @supports (anchor-name:--x)로 감지해 미지원 브라우저엔 일반 절대배치 폴백을 넣었습니다.
  • 툴팁 용도로 쓸 땐 hover만이 아니라 포커스·키보드로도 열리게 하고, 유일한 라벨이면 안 됩니다(WCAG).
paysable 적용: 헤더 계정 메뉴, 거래 행의 "⋯ 더보기", 결제 필드 옆 도움말 툴팁("가맹점 표기가 왜 달라요?"). Floating UI 의존을 걷어내 번들을 줄이고, 스크롤 컨테이너 안에서도 위치가 어긋나지 않습니다. 다만 구형 브라우저 비중이 있는 결제 환경이면 @supports 폴백은 필수(추정: 사용자층에 맞춰 검증).

🔔 폼 에러 요약 — 제출 실패를 한 번에 모아 보여주기 피드백·상태(로딩·빈상태·에러)

긴 결제 폼에서 제출을 눌렀는데 여러 칸이 틀렸을 때, 각 칸 밑에 에러를 흩뿌리기만 하면 사용자는 뭐가 몇 개 틀렸는지 못 봅니다. 에러 요약(error summary) 배너를 폼 맨 위에 띄우고 거기로 포커스를 옮기는 게 접근성 모범 패턴입니다. 빈칸으로 제출해 보세요.

입력을 확인해 주세요

    이름을 입력해 주세요.
    카드 번호 16자리를 정확히 입력해 주세요.
    유효기간을 MM/YY 형식으로 입력해 주세요.

    ✅ 통과! 실제라면 여기서 결제 승인을 진행합니다.

    제출하면 상단 요약 배너로 포커스가 이동합니다. 배너의 각 항목을 클릭하면 해당 입력 칸으로 바로 점프합니다.

    🎯 이건 뭐고, 요즘은 왜 이렇게 하나

    • 흩뿌린 인라인 에러의 한계: 칸 밑 메시지는 그 칸을 볼 때만 유용합니다. 화면 밖으로 스크롤됐거나 개수가 많으면 사용자는 전체 그림을 못 봅니다.
    • 에러 요약 배너: 폼 맨 위에 "3개 항목을 확인하세요" 처럼 모아서 보여주고, 각 항목은 문제 입력으로 가는 링크입니다. GOV.UK·많은 디자인 시스템의 표준 패턴입니다.
    • 포커스 이동이 핵심: 제출 실패 시 요약 배너(tabindex="-1")에 focus()를 줍니다. 스크린리더 사용자는 즉시 "오류가 몇 개 있다"를 듣고, 시각 사용자는 배너가 눈에 들어옵니다.

    💻 코드 (복붙용)

    function onSubmit(e){
  e.preventDefault();
  const errors = validate();           // [{id:'cardNo', msg:'…'}, …]

  // 각 칸 표시
  fields.forEach(f => f.setAttribute('aria-invalid',
     errors.some(er => er.id === f.id)));

  if (errors.length){
    // 요약 배너 채우고 → 포커스 이동 (스크린리더가 즉시 읽음)
    renderSummary(errors);             // <a href="#cardNo">…</a> 목록
    summary.classList.add('show');
    summary.focus();                   // tabindex="-1" 이라 포커스 가능
    return;
  }
  submitPayment();
}

// 요약 링크 클릭 → 해당 입력으로 포커스
list.addEventListener('click', e => {
  const a = e.target.closest('a'); if(!a) return;
  e.preventDefault();
  document.querySelector(a.getAttribute('href')).focus();
});

    ♿ 접근성·상태 설계

    • 배너는 role="alert"(또는 aria-live)로 즉시 알림. 문제 입력엔 aria-invalid="true" + 에러 메시지를 aria-describedby로 연결(이 데모는 인접 요소로 간단화).
    • 색만으로 오류를 표시하지 않기 — 아이콘·텍스트·테두리를 함께(색맹 배려, WCAG 1.4.1).
    • 제출 시점에 요약을, 개별 칸은 사용자가 만진 뒤(blur/입력)에 검증하는 게 덜 성가십니다.
    paysable 적용: 카드 등록·결제 폼, 정산 계좌 등록처럼 칸이 많고 규칙이 빡빡한 폼에 그대로 맞습니다. 서버 검증 실패(예: 카드 거절)도 같은 요약 배너에 얹어 클라이언트·서버 오류를 한 UI로 통일하면 사용자가 헷갈리지 않습니다.

    🌀 height:auto 로 부드럽게 펼치기 — interpolate-size 트렌드·개념

    아코디언·상세 토글을 부드럽게 펼치는 건 오래된 난제였습니다. CSS는 height:0 → auto 전환을 애니메이션 못 해서, 다들 max-height 꼼수나 JS로 픽셀을 재곤 했죠. interpolate-size: allow-keywords가 이걸 진짜로 해결합니다. 아래 항목을 펼쳐보세요.

    아메리카노 ×2 · 4,000원
    크루아상 ×1 · 3,500원
    배송비 · 3,000원
    합계 · 10,500원
    신용/체크카드, 계좌이체, 간편결제를 지원합니다. 할부는 5만원 이상 카드 결제에서 선택할 수 있고, 취소 시 승인 취소로 처리됩니다. 내용 길이가 서로 달라도 각자 제 높이만큼 부드럽게 펼쳐집니다.
    결제 후 7일 이내, 미사용 상품에 한해 전액 환불됩니다. 승인 취소는 카드사 사정에 따라 3~5영업일이 걸릴 수 있습니다.
    확인 중…

    지원 브라우저에선 부드럽게 펼쳐지고, 미지원 브라우저에선 즉시 펼쳐집니다 — 어느 쪽이든 기능은 동일하게 동작(점진적 향상).

    📈 왜 지금 뜨나 / 어디에 쓰나

    • 오래된 통증 해소: height를 auto(내용 크기)로 전환하는 걸 CSS가 못 하던 문제를, interpolate-size: allow-keywords 한 줄이 풀어줍니다. 브라우저가 auto를 실제 픽셀로 환산해 사이 값을 그려 줍니다.
    • calc-size(): auto 같은 키워드로 계산도 가능합니다. 예: calc-size(auto, size - 20px) = "펼친 높이에서 20px 뺀 값". size는 환산된 내부 크기를 가리킵니다.
    • 지원 범위(검색 기준): 현재 Chromium 계열(Chrome·Edge 129+)만 지원, Firefox·Safari 미지원. 그래서 점진적 향상으로 씁니다 — 미지원 브라우저는 애니메이션 없이 바로 펼침(깨지지 않음).

    💻 코드 (복붙용)

    /* 문서 전역에 딱 한 번 켜기 (지원 브라우저에서만) */
@supports (interpolate-size: allow-keywords){
  :root { interpolate-size: allow-keywords; }
}

.panel{
  height: 0;
  overflow: hidden;
  transition: height .3s ease;   /* 이제 auto 로의 전환이 애니메이션됨 */
}
.panel[data-open="true"]{ height: auto; }

/* 모션 최소화 사용자 배려 */
@media (prefers-reduced-motion: reduce){
  .panel{ transition: none; }
}
    <!-- JS는 aria 상태 토글만 -->
<button aria-expanded="false" aria-controls="p1">주문 상세</button>
<div class="panel" id="p1" data-open="false"> … </div>

    ♿ 접근성·상태 설계

    • 헤더 버튼에 aria-expanded(펼침 상태)와 aria-controls(어떤 패널인지)를 연결 — 스크린리더가 "펼침/접힘"을 읽습니다.
    • 애니메이션은 장식일 뿐, 없어도 기능은 그대로. prefers-reduced-motion(모션 최소화 설정)에선 전환을 끕니다.
    • 펼쳐진 내용은 실제 DOM에 있어야(단순히 시각적으로만 숨기지 않기) 검색·포커스가 됩니다.
    paysable 적용: 결제 확인 화면의 "주문 상세 접기/펼치기", 영수증 항목 더보기, FAQ·약관 아코디언. max-height 꼼수는 큰 값을 잡아 두면 타이밍이 어색했는데, interpolate-size는 내용 실제 높이에 맞아 자연스럽습니다. 지금은 점진적 향상으로 얹고, Firefox·Safari가 따라오면 그대로 전면 적용하면 됩니다.
