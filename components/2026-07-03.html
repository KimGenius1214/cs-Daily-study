📊 스파크라인·미니 차트 — 라이브러리 없이 CSS·SVG로 트렌드 보여주기 데이터·테이블·차트

스파크라인(sparkline)은 축·범례 없이 추세선만 작게 그린 미니 차트다. 숫자 카드(KPI) 옆에 붙여 "지금 값이 얼마인가"뿐 아니라 "어느 방향으로 움직여 왔나"를 한눈에 준다. 대시보드 요약 타일엔 Chart.js 같은 무거운 라이브러리가 과하다 — 인라인 SVG 한 줄이나 CSS conic-gradient·flex 막대면 의존성 0으로 충분하다.

막대 하나에 포커스(Tab)를 주거나 마우스를 올리면 값이 뜹니다. 전체 요약은 role="img"+aria-label로 스크린리더에 한 번에 전달.

CSS conic-gradient 도넛 — 승인/실패 비율 같은 단일 지표(값 바꿔보기)
94%
승인 94%
실패·취소 6%

슬라이더를 움직이면 도넛이 갱신됩니다. 도넛은 이미지 하나 없이 conic-gradient() 한 줄로 그립니다.

<!-- SVG 스파크라인: 점들을 polyline 좌표로만 그린다(축·격자 없음) -->
<svg class="spark up" viewBox="0 0 100 32" preserveAspectRatio="none" aria-hidden="true">
  <path class="area" d="M0,20 L9,18 ... L100,6 V32 H0 Z"/>   <!-- 채움 -->
  <path class="line" d="M0,20 L9,18 ... L100,6"/>           <!-- 선 -->
  <circle cx="100" cy="6" r="2.4"/>                        <!-- 마지막 점 강조 -->
</svg>

/* CSS 막대: 값(%)을 height 로, 도넛은 conic-gradient 로 */
.bar .fill { height: var(--h); }
.donut { background: conic-gradient(var(--c) calc(var(--p)*1%), var(--line) 0); }
  • 왜 라이브러리 없이: 요약 타일 수십 개에 각각 차트 인스턴스를 붙이면 번들 크기·렌더 비용이 커진다. 추세만 보여줄 땐 정적 SVG path나 CSS 그라데이션이 훨씬 가볍고, 테마 색은 currentColor·CSS 변수로 자동 대응된다.
  • 접근성: 장식용 미니 차트는 aria-hidden="true"로 숨기고, 의미는 옆의 숫자·델타 텍스트가 전한다. 차트 자체가 정보라면 role="img"+aria-label로 요약을 한 문장 준다.
  • preserveAspectRatio="none" = SVG를 컨테이너 폭에 맞춰 가로로 늘려도 되게 하는 설정(스파크라인은 비율 왜곡이 문제 안 됨). font-variant-numeric:tabular-nums = 숫자 폭을 고정해 값이 바뀌어도 자리가 안 흔들리게.
  • conic-gradient 도넛의 한계: 단일 비율(1개 값) 표현엔 최적이지만, 여러 조각(파이)·툴팁·애니메이션이 필요하면 SVG나 라이브러리가 낫다. @property로 --p를 등록하면 각도 전환도 애니메이션되지만 지원은 브라우저마다 상이(추정).
  • paysable 적용: 정산 대시보드의 "일 거래액·승인율·환불율" 타일에 스파크라인, 요일·시간대 분포에 CSS 막대, 승인율 게이지에 도넛. 무거운 차트는 상세 화면에만, 요약엔 이 미니 차트로.

🔔 토스트 스택 — 실행취소·자동 닫힘·스크린리더 알림까지 피드백·상태(로딩·빈상태·에러)

토스트(toast)는 화면 구석에 잠깐 떴다 사라지는 알림이다. 흐름을 막지 않고 "저장됨 / 삭제됨 / 실패"를 알린다. 핵심 설계 3가지: ① 여러 개가 뜨면 스택으로 쌓고 오래된 건 자동으로 빠진다, ② 파괴적 액션(삭제)엔 실행취소(undo)를 붙여 되돌릴 시간을 준다, ③ 스크린리더엔 aria-live로 읽어 준다. "예쁜 팝업"이 아니라 되돌릴 수 있는 안전장치가 요점.

버튼을 눌러 토스트를 쌓아 보세요. 삭제 토스트의 실행취소는 5초 안에 눌러야 합니다. 자동 닫힘 타이머는 토스트 위에 마우스를 올리면 멈춥니다(읽을 시간 확보).

// 알림 영역: 화면엔 안 보여도 스크린리더가 새 토스트를 읽도록 aria-live
<div class="toast-region" role="region" aria-label="알림">…</div>
<!-- 각 토스트: 긴급도에 따라 role/aria-live 를 나눈다 -->
// 일반 정보 → role="status"  aria-live="polite"  (지금 읽던 걸 끊지 않음)
// 오류·경고 → role="alert"   aria-live="assertive"(즉시 끼어들어 알림)

// 삭제는 "낙관적 UI": 목록에서 먼저 지우고, undo 누르면 되살린다
function deleteRow(row){
  hide(row);                                   // 화면에서 즉시 제거(빠른 반응)
  const t = toast('삭제됨', {undo:()=>show(row), timeout:5000});
  t.onExpire = ()=> commitDelete(row);         // 시간 지나면 진짜 삭제 확정
}
  • 왜 실행취소인가: 삭제 전에 "정말요?" 모달을 띄우면 매번 흐름이 끊긴다. 대신 먼저 실행하고 되돌릴 길을 열어 두는 방식(undo 토스트)이 빠르면서 안전하다. Gmail의 "실행취소"가 대표 사례.
  • 낙관적 UI(optimistic UI) = 서버 응답을 기다리지 않고 성공했다고 가정하고 화면을 먼저 바꾸는 기법. 실패하면 되돌린다. 반응이 즉각적이라 빠르게 느껴진다.
  • aria-live 두 단계: polite(정보)는 사용자가 하던 낭독을 끝낸 뒤 읽고, assertive(오류)는 즉시 끼어든다. 토스트는 키보드로 닫고 undo도 눌러야 하므로 버튼은 실제 <button>으로, 포커스 링을 남긴다.
  • 자동 닫힘의 함정: 너무 빨리 사라지면 못 읽는다. 호버/포커스 시 타이머를 멈추고(prefers-reduced-motion이면 진행바 애니메이션도 끔), 중요한 액션(undo)엔 넉넉한 시간(5초+)을 준다. 오류 토스트는 자동으로 닫지 않는 편이 안전.
  • paysable 적용: "결제 링크 복사됨", "정산 내보내기 완료" 같은 가벼운 성공은 polite 토스트로. "환불 실패(사유)"는 assertive + 자동닫힘 없이. 거래 목록의 행 삭제엔 undo 토스트로 실수를 되돌릴 여지를 남긴다.

🌀 text-wrap: balance / pretty — CSS 한 줄로 줄바꿈 다듬기 트렌드·개념

제목이 두 줄로 넘칠 때 둘째 줄에 단어 하나만 덩그러니 남거나, 문단 끝에 외톨이 단어(orphan)가 떨어지면 지저분해 보인다. 예전엔 <br>를 손으로 넣거나 JS로 처리했지만, 이제 CSS text-wrap: balance(제목용)와 text-wrap: pretty(본문용) 두 값이면 브라우저가 알아서 줄을 고르게 나눠 준다.

켜기 balance(제목) + pretty(본문)

2026년 3분기 결제 정산 리포트가 도착했어요

이번 분기 승인율은 지난 분기 대비 소폭 상승했고, 환불율은 안정적으로 유지되었습니다. 자세한 내역은 대시보드에서 확인하세요.

폭 슬라이더를 좁혀 보세요. 왼쪽(기본)은 둘째 줄에 단어가 몰리거나 끝에 외톨이가 생기고, 오른쪽(balance/pretty)은 줄이 더 고르게 나뉩니다. 지원 안 하는 브라우저에선 그냥 기본 줄바꿈이 되어(깨지지 않음) 안전합니다.

/* 제목: 줄 길이를 고르게 나눔 (짧은 블록 6~10줄 이하에서 동작) */
h1, h2, .headline { text-wrap: balance; }

/* 본문: 마지막 줄의 외톨이 단어(orphan)만 똑똑하게 방지 */
p { text-wrap: pretty; }
  • 둘의 차이: balance는 모든 줄 길이를 비슷하게 맞춰 제목·짧은 블록에 좋다(계산 비용 때문에 브라우저가 몇 줄 이하로 제한). pretty는 문단 끝의 외톨이 단어만 손봐서 긴 본문에 적합. (출처: MDN, WebKit, Chrome for Developers)
  • orphan(외톨이 단어) = 문단 맨 끝 줄에 단어 하나만 남는 것. 미관상 안 좋아 편집·타이포에서 오래 피해 온 대상이다.
  • 지원 현황(2026): balance는 Chrome/Edge 114+, Firefox 121+, Safari 17.5+로 사실상 전부. pretty는 Chrome/Edge 117+, Safari 26+에서 되고 Firefox는 아직 미지원(2026 초 기준, 미지원 시 normal로 폴백). 둘 다 점진적 향상이라 안 되는 곳에선 조용히 기본 줄바꿈으로 떨어진다. (출처: caniuse, MDN)
  • 주의: balance는 줄 수가 많은 긴 본문엔 적용되지 않으므로 제목·리드 문구·버튼 라벨 같은 짧은 텍스트에 쓴다. 성능 걱정으로 문단 전체에 balance를 남발하지 말 것.
  • paysable 적용: 카드/타일 제목, 빈 상태 안내 문구, 결제 실패 사유 메시지처럼 폭이 가변인 짧은 텍스트에 balance. 약관·안내 본문엔 pretty. 반응형에서 폭이 계속 바뀌는 결제 UI일수록 손으로 <br> 넣던 수고가 사라진다.
