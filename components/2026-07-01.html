⌨️ 인증코드(OTP) 입력 — 칸 분리 · 붙여넣기 · 자동 이동 입력·폼

OTP(One-Time Password, 한 번만 쓰는 인증코드)는 결제 3DS 인증·로그인에서 자주 쓰인다. 한 칸짜리 입력보다 자릿수만큼 칸을 나눈 형태가 "몇 자리인지" 한눈에 보이고 오타를 줄인다. 관건은 손맛이다: 한 글자 치면 다음 칸으로 자동 이동, 지우면 이전 칸, 문자 메시지에서 복사한 6자리를 아무 칸에나 붙여넣으면 알아서 분배되어야 한다.

데모 정답은 424242. 한 칸씩 입력하거나, 어느 칸이든 424242를 통째로 붙여넣어 보세요.

  • inputmode="numeric" = 모바일에서 숫자 키패드를 띄운다. autocomplete="one-time-code" = iOS Safari가 방금 온 SMS의 코드를 키보드 위에 자동완성으로 제안한다(추정: 안드로이드 크롬은 별도의 WebOTP API를 써야 자동 채움).
  • 붙여넣기 처리가 핵심: 6자리를 한 칸에 붙여도 paste 이벤트를 가로채 칸마다 나눠 넣는다. 키보드 사용자는 ← → 백스페이스로 칸을 오갈 수 있어야 한다.
  • paysable 적용: 결제 승인 3DS·본인확인 코드 입력 화면. 자동 이동·붙여넣기·재전송 타이머(쿨다운)까지 갖추면 입력 이탈이 크게 준다.

💳 결제 수단 선택 — 라디오 카드 그룹 결제·체크아웃 UX

결제 수단 고르기는 본질이 라디오 버튼(여럿 중 하나)이다. 하지만 작은 동그라미 대신 카드 전체를 누르는 영역으로 키우고, 아이콘·이름·설명을 함께 보여주면 손가락으로 누르기 쉽고 비교가 빠르다. 겉모습만 바꾸되 접근성은 진짜 라디오로 유지하는 게 요령이다 — 그래야 키보드 화살표로 넘기고 스크린리더가 "라디오, 2/4 선택됨"으로 읽는다.

선택: 신용·체크카드

카드 아무 곳이나 클릭하면 선택됩니다. 키보드라면 Tab으로 그룹에 들어와 ← → ↑ ↓로 수단을 바꿔보세요.

  • 구조: 보이지 않게 숨긴 <input type="radio">를 <label>로 감싸 카드 전체를 클릭 영역으로 만든다. 선택 표시는 CSS :has(input:checked)로 그린다(JS 없이도 동작).
  • :has() = "이런 자식을 가진 부모"를 고르는 CSS 선택자. 체크된 라디오를 품은 카드를 강조할 때 쓴다(2026 기준 주요 브라우저 지원).
  • 접근성: 시각적으로 카드를 써도 실제 요소가 라디오라 화살표 키 이동·그룹 읽기가 공짜로 따라온다. role="radiogroup"로 묶어 묶음임을 알린다.
  • paysable 적용: 체크아웃의 결제수단 선택, 배송 옵션, 요금제 선택 등 "여럿 중 하나" 화면 전반에.

🌀 :user-valid / :user-invalid — '사용자가 만진 뒤에만' 검증 스타일 트렌드·개념

기존 :valid/:invalid는 페이지가 뜨자마자 빈 필수 칸을 빨갛게 칠해버려 "아직 입력도 안 했는데 에러" 같은 짜증을 만들었다. 2026년엔 :user-valid/:user-invalid를 쓴다. 이 둘은 사용자가 실제로 입력하고 칸을 벗어난 뒤에만 적용돼서, JavaScript 없이도 "건드린 칸만 검증"이 된다.

:user-invalid — 처음엔 조용, 잘못 입력하고 벗어나야 빨갛게
올바른 이메일 형식이면 초록, 틀리면 빨강으로 바뀝니다. 이메일 형식이 올바르지 않습니다.
16자리를 정확히 채우면 초록으로 바뀝니다. 숫자 16자리를 입력하세요.
옛 방식 :invalid — 만지기도 전에 이미 빨강 (이게 짜증의 원인)
페이지를 열자마자 빨간 테두리죠? 사용자는 아무 잘못도 안 했는데요.
/* 사용자가 입력하고 칸을 벗어난 뒤에만 검증 색을 입힌다 */
input:user-invalid { border-color: var(--danger); }
input:user-valid   { border-color: var(--ok); }

/* 옛 방식: 로드 즉시 빈 필수 칸까지 빨갛게 (지양) */
input:invalid { border-color: red; }
  • 왜 지금: :user-valid/:user-invalid가 Baseline(주요 브라우저 모두 지원)이 됐다 — Chrome·Edge·Firefox·Safari, 2023년 말부터 폭넓게 사용 가능.
  • 차이 한 줄: :invalid는 "값이 규칙에 안 맞으면 즉시", :user-invalid는 "사용자가 의미 있게 상호작용한 뒤에만". UX 체감이 크게 다르다.
  • 구형 브라우저면 검증 색만 생략될 뿐 입력 자체는 정상 동작한다(graceful degradation = 점진적 기능 축소).
  • paysable 적용: 결제·가입 폼의 인라인 검증을 JS 상태(touched 플래그) 없이 CSS만으로 처리 → 코드·번들 감소. 단, 제출 시 종합 에러 안내는 여전히 코드로 보완.
