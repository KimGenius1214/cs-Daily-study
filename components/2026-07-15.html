💳 금액 입력 필드 결제·체크아웃 UX

돈을 넣는 칸은 일반 텍스트 칸과 다르다. 천 단위 콤마를 타이핑하는 동안 붙여주고, 모바일에선 숫자 키패드가 떠야 하고, 한도를 넘으면 즉시 막아야 한다. 아래 4가지 변형을 직접 눌러보자.

변형 A — 기본형 (자동 콤마 + 통화 기호)

₩

숫자만 입력 — 콤마는 자동으로 붙습니다.

왜: inputmode="numeric"은 모바일에서 숫자 키패드를 띄운다(type="number"는 스피너 화살표·스크롤 오작동 때문에 금액엔 잘 안 쓴다). 커서 위치를 계산해 되돌려주므로 중간을 고쳐도 안 튄다.

변형 B — 한도 검증 + 잔액 표시

₩
사용 가능 잔액1,250,000원

왜: 잔액을 입력칸 바로 밑에 두면 "얼마까지 되나"를 찾으러 이동할 필요가 없다. 에러는 role="alert"로 스크린리더에 즉시 읽히고, 넘으면 제출 버튼이 잠긴다.

변형 C — 빠른 금액 칩

₩

왜: 실제 충전액은 몇 가지에 몰린다. 칩은 더하기로 두면(설정이 아니라) 5만+1만=6만 같은 조합을 탭 두 번에 만든다.

변형 D — 커스텀 키패드 (앱 스타일)

₩ 0

왜: OS 키보드가 화면을 반쯤 가리는 걸 피하고 싶을 때 쓴다. 대신 진짜 버튼으로 만들어야 키보드·스크린리더로도 조작된다(div에 onclick 달면 접근 불가).

용어 — tabular-nums: 숫자 폭을 모두 같게 맞추는 폰트 기능. 켜두면 금액이 바뀔 때 자릿수가 좌우로 덜컹거리지 않는다. 위 모든 금액 칸에 font-variant-numeric:tabular-nums로 적용돼 있다.

💻 핵심 코드 — 커서 안 튀는 자동 콤마

const fmt = new Intl.NumberFormat('ko-KR'); // 1234567 → "1,234,567"

function onAmountInput(e) {
  const el = e.target;
  // 1) 커서 왼쪽에 숫자가 몇 개였는지 세둔다 (콤마는 무시)
  const digitsLeft = el.value.slice(0, el.selectionStart).replace(/\D/g, '').length;

  // 2) 숫자만 남겨 포맷
  const digits = el.value.replace(/\D/g, '').slice(0, 12); // 자릿수 상한
  el.value = digits ? fmt.format(Number(digits)) : '';

  // 3) 같은 "숫자 개수" 지점으로 커서 복원 → 중간 수정해도 끝으로 안 튐
  let pos = 0, seen = 0;
  while (pos < el.value.length && seen < digitsLeft) {
    if (/\d/.test(el.value[pos])) seen++;
    pos++;
  }
  el.setSelectionRange(pos, pos);
}

순진하게 value = format(value)만 하면 콤마가 붙는 순간 커서가 맨 끝으로 날아간다. 위처럼 숫자 개수 기준으로 복원해야 자연스럽다.

⌨️ 날짜·기간 선택 입력·폼

정산 내역, 거래 조회, 쿠폰 유효기간 — 결제 서비스는 기간 선택이 사방에 있다. 요즘 기본 원칙: 네이티브를 먼저 쓰고, 프리셋으로 대부분을 없애고, 캘린더는 최후에.

왜: 시작일을 고르면 종료일의 min을 그 날짜로 즉시 올린다. "끝이 시작보다 빠릅니다" 에러를 띄우는 대신 애초에 못 고르게 하는 게 낫다. 네이티브 input[type=date]는 공짜로 로케일·키보드·모바일 휠 UI를 준다.

변형 B — 프리셋 칩이 먼저

–

왜: 정산 화면 요청의 대다수는 "최근 7일/이번 달"이다. 칩을 위에 두면 클릭 1번으로 끝나고, 날짜칸은 예외용으로 남는다. 칩을 누르면 아래 칸도 같이 채워져 "무엇이 선택됐는지"가 투명하다.

🌀 light-dark() — 한 줄에 라이트·다크 색을 같이 트렌드·개념

지금까지 다크 모드는 "변수 세트를 두 벌 만들고 미디어 쿼리로 갈아끼우기"였다. light-dark()는 한 선언 안에 두 색을 나란히 적는다. 이 카드 안의 아래 박스들은 실제로 그렇게 만들어져 있다.

용어 — color-scheme: 이 요소가 라이트/다크 중 무엇으로 그려질지 브라우저에 알리는 CSS 속성. light-dark()는 이 값을 보고 둘 중 하나를 고른다. 그래서 color-scheme을 안 켜면 light-dark()는 항상 첫 번째 값을 쓴다 — 가장 흔한 실수다.

직접 전환 — 페이지 테마와 무관하게 강제

light-dark()로만 칠한 박스

배경·글자·테두리·칩 색이 전부 light-dark(라이트값, 다크값) 한 줄씩입니다. 미디어 쿼리 없음, JS로 클래스 갈아끼우기 없음 — 위 버튼은 color-scheme 한 개만 바꿉니다.

📈 왜 지금 뜨나

브라우저 지원이 다 채워졌다. Chrome 123(2024-03), Firefox 120(2023-11), Safari 17.5(2024-05)에서 지원되며, 2024년 5월부터 최신 브라우저 전 기종에서 동작한다. Baseline 기준으로는 현재 "newly available", 2026년 11월 13일에 "widely available"로 승격 예정이다(MDN·web-features 기준). 즉 지금 쓰면 대부분 커버되지만, 아주 오래된 브라우저를 챙기는 서비스라면 폴백을 같이 두는 게 안전하다.

💻 코드 — 기존 방식 vs light-dark()

/* 이전: 변수 두 벌 + 미디어 쿼리 + 수동 토글 클래스 */
:root { --bg:#fff; --text:#14181f; }
@media (prefers-color-scheme: dark) {
  :root { --bg:#171b21; --text:#eef1f5; }
}
[data-theme="dark"] { --bg:#171b21; --text:#eef1f5; }  /* 수동 토글용 중복 */

/* 이후: 한 줄에 두 색. 스위치는 color-scheme 하나 */
:root {
  color-scheme: light dark;          /* ← 이게 없으면 항상 첫 번째 값이 나온다 */
  --bg:   light-dark(#ffffff, #171b21);
  --text: light-dark(#14181f, #eef1f5);
}
.force-light { color-scheme: light; } /* 부분 강제도 요소 단위로 */
.force-dark  { color-scheme: dark; }

/* 폴백이 필요하면: 앞줄이 구형 브라우저용, 뒷줄을 지원 브라우저가 덮어쓴다 */
.box { background:#ffffff; background: light-dark(#ffffff, #171b21); }

이 방식의 장단점

좋은 점
· 색 정의가 한 곳에 모인다 — 라이트만 고치고 다크를 까먹는 사고가 준다.
· 미디어 쿼리 블록·중복 선택자가 사라져 CSS가 짧아진다.
· color-scheme은 상속되므로, 특정 영역만 다크로 뒤집는 게 클래스 하나로 끝난다.
· 네이티브 위젯(스크롤바, input[type=date] 달력, 기본 폼 컨트롤)도 color-scheme을 따라 같이 어두워진다 — 원래도 이 속성의 본래 역할이다.

주의할 점
· 두 가지만 된다. "고대비" 같은 3번째 테마가 있으면 결국 변수 세트가 필요하다.
· 각 색이 어디 쓰이는지 이름으로만 보이던 게, 값 두 개로 늘어 줄이 길어진다.
· 색 이외 값(그림자 세기, 보더 두께)은 못 담는다 — light-dark()는 색·이미지 전용.
· 추정: 아주 큰 팔레트에서 성능 차이는 없을 것으로 보이나 측정해본 바 없음.

참고로 이 페이지 나머지 부분은 일부러 예전 방식([data-theme] 변수 세트)으로 두었다. 위 박스 하나만 light-dark()다 — 같은 화면에서 두 방식이 섞여도 문제없다는 것도 같이 보여주려는 의도다.
