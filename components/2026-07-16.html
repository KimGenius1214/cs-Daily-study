대시보드

좌측 메뉴 접기를 눌러보세요. 접히면 라벨이 사라지고, hover/키보드 포커스 시 툴팁으로 이름이 나옵니다. 배지(12)는 접힘 상태에선 점(dot)으로 축약됩니다.

Tab 키로만 이동해봐도 모든 항목에 포커스 링이 보이고, 접힌 상태에서도 이름을 알 수 있습니다.

왜 이렇게 하나 — 접힘 상태에서 아이콘만 남기면 이게 뭔지 알 수 없어진다. 그래서 툴팁이 필수다. 툴팁을 hover에만 걸면 키보드 사용자는 못 보므로 :focus-visible에도 같이 건다.

배지를 왜 점으로 줄이나 — 64px 폭에 "12"를 우겨넣으면 아이콘과 겹친다. 접힘 상태의 목적은 "무언가 있다"는 신호만 남기는 것이고, 정확한 숫자는 펼쳤을 때 보면 된다.

변형 B — 브레드크럼(빵부스러기)

사이드바가 어디로 갈 수 있나를 보여준다면, 브레드크럼은 지금 어디에 있나 + 한 단계 위로를 보여준다. 위 데모의 본문 상단(홈 / 가맹점 / 대시보드)이 그것이다. 마지막 항목은 링크가 아니라 aria-current="page"가 붙은 텍스트여야 한다 — 지금 있는 곳으로 가는 링크는 무의미하다.

💻 코드 (복붙용)

<!-- 접힘 상태를 컨테이너의 data 속성 하나로 제어 -->
<div class="app" data-collapsed="false">
  <nav class="side" aria-label="주 메뉴">
    <button aria-expanded="true" aria-controls="appShell">☰ <span class="label">메뉴 접기</span></button>
    <a class="navitem" href="/dashboard" aria-current="page">
      <span class="ico" aria-hidden="true">📊</span><span class="label">대시보드</span>
    </a>
  </nav>
</div>

<style>
.side{ width:232px; transition:width .22s ease }
.app[data-collapsed="true"] .side{ width:64px }
.app[data-collapsed="true"] .label{ opacity:0; width:0 }        /* 라벨 숨김 */

/* 활성 표시: 색 + 좌측 바 + aria-current (3중) */
.navitem[aria-current="page"]{ background:var(--brand-soft); color:var(--brand); font-weight:700 }
.navitem[aria-current="page"]::before{
  content:""; position:absolute; left:-10px; top:8px; bottom:8px;
  width:3px; border-radius:0 3px 3px 0; background:var(--brand);
}

/* 접힘 상태에서만 툴팁 — hover와 focus-visible 둘 다 */
.app[data-collapsed="true"] .navitem:hover .label,
.app[data-collapsed="true"] .navitem:focus-visible .label{
  opacity:1; width:auto; position:absolute; left:calc(100% + 8px);
  top:50%; transform:translateY(-50%);
  background:var(--text); color:var(--bg); padding:4px 9px; border-radius:6px;
  z-index:5; pointer-events:none;
}
</style>

<script>
// Next.js라면 usePathname()으로 aria-current 계산
// const active = pathname === href ? 'page' : undefined;
btn.addEventListener('click', () => {
  const collapsed = app.dataset.collapsed === 'true';
  app.dataset.collapsed = String(!collapsed);
  btn.setAttribute('aria-expanded', String(collapsed));
});
</script>

♿ 접근성 · 상태 설계

  • <nav aria-label="주 메뉴"> — 한 페이지에 nav가 여럿이면(주 메뉴, 브레드크럼, 푸터) 각각 라벨을 줘야 스크린리더가 구분한다.
  • 접힘 토글은 aria-expanded — 버튼이 무언가를 펼치고 접는다는 사실 자체를 알린다. aria-controls로 대상 id를 가리킨다.
  • 중첩 그룹은 <button aria-expanded> + hidden — 접혀 있는 하위 항목은 hidden으로 접근성 트리에서도 빼야 한다. display:none 없이 height:0만 주면 보이지 않는데 Tab이 걸리는 버그가 난다.
  • 포커스 링 — :focus-visible은 "키보드로 이동했을 때만" 링을 보여주는 선택자. 마우스 클릭 시엔 링이 안 뜬다.
  • 사이드바 접힘 상태는 기억 — 새로고침해도 유지되게 서버 쿠키나 사용자 설정에 저장(이 데모는 메모리에만).

🧠 paysable 적용 포인트

  • 거래내역 같은 넓은 테이블 화면에서 사이드바를 접으면 가로 168px을 벌 수 있다 — 컬럼 2개 분량.
  • 배지는 "처리 대기 정산 12건" 같은 액션 필요 알림에만. 단순 카운트에 빨간 배지를 쓰면 알림 피로가 생긴다.
  • Next.js App Router라면 usePathname()으로 aria-current를 계산하되, 정확 일치 vs 접두사 일치를 정해둘 것. /settlements와 /settlements/123에서 둘 다 활성이어야 하는지 결정 필요.

🃏 선택형 리스트 · 벌크 액션 바 부분 선택 · 컨텍스트 액션 · :has()

"거래 3건 골라서 한 번에 내보내기". 목록에서 여러 항목을 고르고 일괄 처리하는 패턴. 핵심은 액션을 미리 보여주지 않고, 선택했을 때만 띄우는 것과 전체선택 체크박스의 세 번째 상태다.

indeterminate(부분 선택) — 체크박스의 숨은 3번째 상태. 체크됨/안 됨 말고 "일부만 선택됨"을 나타내는 줄표(–) 모양. HTML 속성이 아니라 JS로만 켤 수 있다: el.indeterminate = true.

변형 A — 거래내역 다중 선택 + 하단 벌크 바

가맹점 / 일시
금액
0건 선택

체크해보세요 — 하단 바가 선택이 있을 때만 올라오고, 선택 건수와 합계 금액을 같이 보여줍니다. 일부만 고르면 상단 전체선택은 줄표(–) 상태가 됩니다. Shift 없이도 각 행 전체가 클릭 영역(<label>)입니다.

왜 바를 상시 노출하지 않나 — 액션 버튼이 항상 떠 있으면 "무엇에 적용되는 건지" 모호하다. 선택했을 때만 나타나면 선택 → 액션이라는 인과가 눈에 보인다. 또 바에 건수를 반드시 적는다 — "결제 취소"를 누를 때 3건인지 300건인지 모르면 사고 난다.

변형 B — 빈 상태(선택할 게 없을 때)

🔍
조건에 맞는 거래가 없습니다
기간을 넓히거나 필터를 지워보세요.

빈 상태에는 다음 행동 버튼을 반드시 준다. "없습니다"로 끝내면 사용자는 막힌다.

💻 코드 (복붙용)

<!-- 행 전체가 클릭 영역: label로 감싼다 -->
<label class="lrow" for="tx1">
  <input type="checkbox" class="tx" id="tx1" aria-label="배달의민족 23,500원 선택">
  ...
</label>

<style>
/* :has() — "체크된 input을 가진 행"을 CSS만으로 선택 (JS 클래스 토글 불필요) */
.lrow:has(input:checked){ background:var(--brand-soft) }
.lrow:has(input:focus-visible){ outline:2px solid var(--ring); outline-offset:-2px }

/* 벌크 바: visibility까지 같이 꺼야 숨겨진 버튼에 Tab이 안 걸린다 */
.bulkbar{ opacity:0; visibility:hidden; transform:translateY(12px);
          transition:opacity .18s, transform .18s, visibility .18s }
.bulkbar[data-show="true"]{ opacity:1; visibility:visible; transform:translateY(0) }
</style>

<script>
const all = document.querySelectorAll('.tx');
function sync(){
  const picked = [...all].filter(c => c.checked);
  // ① 전체선택 체크박스의 3가지 상태
  selAll.checked = picked.length === all.length && picked.length > 0;
  selAll.indeterminate = picked.length > 0 && picked.length < all.length;
  // ② 선택이 있을 때만 바를 띄우고, 건수·합계를 표시
  bulkBar.dataset.show = String(picked.length > 0);
  bulkCnt.textContent = `${picked.length}건 선택`;
  // ③ 스크린리더에도 알림 (aria-live="polite" 영역)
  liveSel.textContent = picked.length ? `${picked.length}건 선택됨` : '선택 해제됨';
}
all.forEach(c => c.addEventListener('change', sync));
selAll.addEventListener('change', () => {
  all.forEach(c => c.checked = selAll.checked);
  sync();
});
</script>
:has() — "이런 자식을 가진 부모"를 고르는 CSS 선택자. 예전엔 JS로 부모에 클래스를 붙여야 했던 걸 CSS 한 줄로 해결한다. (2026-07 기준 주요 브라우저 지원 — Baseline "widely available", MDN 기준)
aria-live="polite" — 화면 안 내용이 바뀌었을 때 스크린리더가 지금 읽던 걸 끊지 않고 이어서 읽어주게 하는 영역. 벌크 바처럼 시각적으로만 나타나는 변화를 소리로도 전달할 때 쓴다.

♿ 접근성 · 상태 설계

  • 숨긴 바는 visibility:hidden까지 — opacity:0만 주면 보이지 않는 버튼에 Tab이 걸린다. 가장 흔한 사고.
  • 각 체크박스에 aria-label — "체크박스"라고만 읽히면 뭘 고르는지 모른다. 행의 요약(가맹점 + 금액)을 넣는다.
  • indeterminate는 시각 표시일 뿐 — 클릭하면 checked/unchecked만 오간다. "일부 선택 상태에서 누르면 전체 선택인가 전체 해제인가"를 정해야 한다(보통 전체 선택).
  • 파괴적 액션엔 확인 — "결제 취소"처럼 되돌리기 어려운 건 건수를 다시 적은 확인 모달을 띄우거나, 실행 후 실행취소 토스트를 준다.
  • 페이지 이동 시 선택 유지 여부를 정할 것. 유지한다면 "다른 페이지 2건 포함 총 5건 선택"처럼 명시해야 한다.

  • 정산 화면의 일괄 승인 / 일괄 보류 / CSV 내보내기에 그대로 대응된다.
  • 합계 금액을 바에 같이 띄우는 게 결제 도메인에선 특히 유용 — "3건 선택 · 합계 156,300원"을 보고 실수를 잡는다.

🌀 스크롤 기반 애니메이션 animation-timeline: scroll() / view()

스크롤 위치에 따라 애니메이션이 진행되는 걸 JS 없이 CSS만으로 만든다. 예전엔 IntersectionObserver나 스크롤 이벤트로 직접 계산했던 것.

animation-timeline — 애니메이션의 "시계"를 시간 대신 스크롤 진행률로 바꾸는 CSS 속성. scroll()은 스크롤 컨테이너가 얼마나 스크롤됐나, view()는 이 요소가 화면에 얼마나 들어왔나를 0~100%로 매핑한다.

변형 A — 읽기 진행바 (scroll())

아래로 스크롤해보세요. 위 진행바가 스크롤 진행률을 따라갑니다.
이 진행바는 JS 이벤트 리스너가 하나도 없습니다.
CSS scroll-timeline: --box y + animation-timeline: --box 두 줄이 전부.
스크롤 이벤트로 구현하면 메인 스레드에서 매 프레임 계산이 돕니다.
스크롤 기반 애니메이션은 컴포지터에서 돌 수 있어 더 부드럽습니다. (일반적으로 그렇게 알려짐 — 실측 필요)
거의 다 왔습니다.
끝. 진행바가 100%입니다.

왜 이렇게 하나 — 문서 읽기 진행바, 긴 폼의 완료율, 가로 캐러셀 인디케이터 등에 쓴다. 스크롤 이벤트 → setState → 리렌더 사슬이 통째로 사라진다.

변형 B — 뷰에 들어올 때 등장 (view())

첫 번째 카드
스크롤을 내리면 각 카드가 화면에 들어오는 만큼 서서히 나타납니다.
이건 animation-timeline: view() 한 줄입니다.
IntersectionObserver + 클래스 토글이 필요 없습니다.
그리고 스크롤을 되돌리면 거꾸로 재생됩니다 — 애니메이션이 아니라 스크롤에 매인 상태이기 때문.
마지막 카드

animation-range: entry 10% cover 32% — "요소가 뷰포트에 10% 들어온 시점부터 32% 지점까지" 애니메이션을 진행시킨다. 범위를 안 주면 요소가 화면을 완전히 통과할 때까지 늘어져서 어색하다.

⚠️ 이 브라우저는 animation-timeline을 지원하지 않아 폴백(최종 상태 고정)으로 표시 중입니다. — @supports not 폴백이 동작한 것.

💻 코드 (복붙용)

/* ① 스크롤 진행바 — 특정 컨테이너의 스크롤에 매기 */
.scroller{ overflow:auto; scroll-timeline: --box y; }   /* 타임라인 이름 짓기 */
.progressbar{
  transform-origin: 0 50%;
  animation: growX linear both;
  animation-timeline: --box;                            /* 그 타임라인에 매기 */
}
@keyframes growX{ from{ transform:scaleX(0) } to{ transform:scaleX(1) } }

/* 문서 전체 스크롤이면 이름 없이: animation-timeline: scroll(root block); */

/* ② 뷰에 들어올 때 등장 */
.reveal{
  animation: fadeUp linear both;
  animation-timeline: view();            /* 이 요소가 화면에 들어온 정도 */
  animation-range: entry 10% cover 32%;  /* 언제 시작해서 언제 끝낼지 */
}
@keyframes fadeUp{
  from{ opacity:0; transform:translateY(14px) scale(.97) }
  to  { opacity:1; transform:none }
}

/* ③ 폴백 — 미지원 브라우저에선 최종 상태로 고정 (내용이 안 보이면 안 되니까) */
@supports not (animation-timeline: view()){
  .reveal{ animation:none; opacity:1; transform:none }
}

/* ④ 모션 최소화 설정 존중 */
@media (prefers-reduced-motion: reduce){
  .reveal{ animation:none; opacity:1; transform:none }
}

📈 왜 지금 뜨나 / 어디에 쓰나

  • 스크롤 애니메이션은 원래 JS의 영역이었다 — AOS, GSAP ScrollTrigger, Framer Motion 같은 라이브러리를 쓰거나 IntersectionObserver를 직접 짰다. 그게 CSS 속성 두 개로 내려왔다.
  • 번들 크기 — 등장 애니메이션 하나 때문에 애니메이션 라이브러리를 넣던 걸 없앨 수 있다.
  • 지원 현황 (2026-07 기준, 확인됨): Chrome/Edge/Safari에서 동작하지만 Firefox는 아직 기본 활성화가 아니라(플래그 뒤) Baseline이 아니다. 즉 있으면 좋은 장식으로만 쓰고, 없으면 안 보이는 필수 콘텐츠에 걸면 안 된다 — 그래서 @supports not 폴백이 필수.
  • 쓰기 좋은 곳: 읽기 진행바, 긴 약관·명세서의 스크롤 위치 표시, 랜딩 페이지 섹션 등장, 가로 캐러셀 인디케이터.
  • 쓰면 안 되는 곳: 실제 데이터가 담긴 목록의 등장 효과(미지원 브라우저에서 문제 나기 쉽고, 스크롤할 때마다 재생돼 피로하다), 접근성 필수 정보.

♿ 접근성 · 상태 설계

  • prefers-reduced-motion — 사용자가 OS에서 "동작 줄이기"를 켰는지 알려주는 미디어 쿼리. 전정기관 장애가 있는 사용자에겐 스크롤 연동 움직임이 어지럼증을 유발할 수 있다. 반드시 대응.
  • 폴백은 "최종 상태"로 — opacity:0이 시작값인데 애니메이션이 안 돌면 콘텐츠가 영영 안 보인다. 미지원 시엔 무조건 보이게.
  • 스크롤 컨테이너에 tabindex="0"과 aria-label — 키보드로도 스크롤할 수 있어야 한다(이 데모도 그렇게 함).

🧠 paysable 적용 포인트

  • 긴 결제 명세서/약관 화면의 상단 진행바 — "어디까지 읽었나"를 JS 없이. 특히 약관 동의 전 스크롤 유도에 시각 피드백으로 유용(단, 실제 "끝까지 읽음" 판정은 여전히 JS로 해야 함 — 애니메이션은 판정 수단이 아님).
  • 랜딩/온보딩 페이지의 섹션 등장 — 지금 라이브러리를 쓰고 있다면 걷어낼 후보.
  • 거래내역 목록에는 쓰지 말 것 — 스크롤을 되돌릴 때마다 카드가 다시 사라졌다 나타나면 데이터 화면에선 산만하다.
