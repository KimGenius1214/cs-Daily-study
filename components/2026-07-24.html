📊 진행률·게이지 — 한도 소진·정산 단계를 CSS·SVG로 데이터·테이블·차트

진행률 표시(progress indicator)는 "얼마나 찼는지 / 어디까지 왔는지"를 한눈에 보여 주는 시각 요소입니다. 결제 도메인에선 카드 한도 소진, 정산 진행 단계, 목표 달성률 같은 데 매일 씁니다. 차트 라이브러리 없이 가로 바·단계 미터·원형 게이지 세 가지를 CSS와 SVG만으로 만들었습니다. 아래 슬라이더를 움직이면 세 표현이 임계값(warning/danger)에 따라 색까지 함께 바뀝니다.

가로 바·게이지는 0~70% 브랜드색 · 70~90% 주황(warn) · 90%+ 빨강(danger)으로 임계값을 색으로도 알립니다(색만으로 판단하지 않도록 옆에 상태 글자·숫자를 함께 둡니다).

🎯 이건 뭐고, 요즘은 왜 이렇게 하나

  • 세 가지 표현, 다른 쓰임새: 가로 바는 연속 비율(한도 소진·업로드)에, 단계 미터는 이산적인 몇 단계(정산 요청→검토→승인→송금→완료)에, 원형 게이지는 목표 대비 달성률처럼 "요약 한 숫자"를 도드라지게 보일 때 씁니다.
  • 라이브러리 없이: 가로 바는 width%, 단계 미터는 flex 칸, 게이지는 SVG 원 하나의 stroke-dasharray/offset으로 그립니다. Chart.js·D3 같은 무거운 의존성 없이 수 KB로 끝나고, 색·굵기를 토큰(CSS 변수)으로 자유롭게 맞춥니다.
  • 임계값을 색으로: 결제·한도는 "얼마 남았나"가 중요하므로 70%/90% 같은 경계에서 색을 바꿔 위험을 미리 알립니다. 단, 색만으로 알리면 색각 사용자가 놓치므로 숫자·상태 글자를 항상 곁들입니다.

💻 코드 (복붙용)

<!-- 원형 게이지: SVG 원 하나로. r=45 → 둘레 2·π·45 ≈ 283 -->
<svg class="gauge" viewBox="0 0 100 100" style="--val:64">
  <circle class="track" cx="50" cy="50" r="45"/>
  <circle class="fill"  cx="50" cy="50" r="45"/>
</svg>
.gauge .fill{
  fill:none; stroke:var(--brand); stroke-width:11; stroke-linecap:round;
  stroke-dasharray:283;                    /* 둘레 전체 */
  stroke-dashoffset:calc(283 - 283 * var(--val) / 100); /* 값만큼만 채움 */
  transform:rotate(-90deg); transform-origin:50% 50%;   /* 12시부터 시작 */
  transition:stroke-dashoffset .6s ease;
}
/* 가로 바는 자식 span 의 width% 만 바꾸면 됨 */
.bar > span{ width:64%; transition:width .5s ease; }

♿ 접근성·상태 설계

  • 진짜 진행률엔 role="progressbar": 결정적 진행(파일 업로드 등)은 <div role="progressbar" aria-valuenow aria-valuemin aria-valuemax> 또는 네이티브 <progress>를 써서 낭독기가 "64%"를 읽게 합니다. 순수 장식 게이지엔 aria-label로 요약을 줍니다.
  • 색 + 텍스트 이중 표기: warn/danger를 색으로만 두지 말고 "위험 · 92%"처럼 글자·숫자를 함께 둡니다(:focus·색각·흑백 출력 대비).
  • 모션 최소화: 채워지는 transition은 prefers-reduced-motion(모션 줄이기 OS 설정)에서 짧게/없이 처리해도 정보 전달엔 지장 없게 합니다.
paysable 적용: 카드·계좌 한도 소진 바, 정산 단계 미터(요청→검토→승인→송금→완료), 대시보드 상단 월 목표 게이지에 바로 씁니다. 이미 다룬 스파크라인·통계 카드와 한 세트로 묶으면 라이브러리 없이도 가벼운 대시보드가 완성됩니다.

🔔 토스트·스낵바 — 스택·자동 소멸·되돌리기(Undo) 피드백·상태(로딩·빈상태·에러)

토스트(toast) / 스낵바(snackbar)는 화면 구석에 잠깐 떴다 사라지는 비차단(non-blocking) 알림입니다. 모달과 달리 사용자의 작업을 막지 않죠. 결제·정산 앱에선 "저장됨 / 전송 실패 / 삭제됨 — 되돌리기"에 씁니다. 아래 버튼으로 여러 개를 쌓아 보고, 실패 토스트의 되돌리기·자동 소멸 타이머를 확인하세요.

여기 우하단에 토스트가 쌓입니다. 최대 3개까지 보이고, 넘치면 가장 오래된 게 밀려납니다.

각 토스트는 약 4초 뒤 자동으로 사라지고(하단 진행 바가 남은 시간), 마우스를 올리면 타이머가 멈춥니다. "되돌리기"가 있는 토스트는 실행을 잠시 미뤄 두었다가, 되돌리면 취소합니다(optimistic 삭제).

🎯 이건 뭐고, 요즘은 왜 이렇게 하나

  • 토스트 vs 모달·인라인: 결과를 알리기만 하면 토스트, 확인·선택을 강제해야 하면 모달, 특정 입력의 잘못이면 그 필드 옆 인라인 메시지가 맞습니다. 토스트에 "정말 삭제?" 같은 중요한 결정을 담지 마세요(놓치면 끝).
  • 되돌리기(Undo) 패턴: "삭제했습니다 — 되돌리기"는 삭제를 즉시 UI에 반영(optimistic UI: 서버 응답을 기다리지 않고 성공을 가정해 먼저 화면을 바꾸는 방식)하되, 실제 실행은 토스트가 살아 있는 몇 초간 미뤄 둡니다. 그동안 되돌리면 아무 일도 없던 것처럼 취소됩니다.
  • 스택·개수 제한: 여러 알림이 동시에 나면 세로로 쌓되 최대 3~4개까지만 두고 오래된 것부터 밀어냅니다. 무한정 쌓이면 화면을 덮어 오히려 방해가 됩니다.

💻 코드 (복붙용)

<!-- 낭독기가 자동으로 읽도록: 컨테이너에 aria-live -->
<div id="stack" role="region" aria-label="알림" aria-live="polite"></div>
function toast({title, msg, type='info', undo}) {
  const el = document.createElement('div');
  el.className = 'toast ' + type;
  el.setAttribute('role', type === 'err' ? 'alert' : 'status'); // 에러는 즉시 낭독
  // ...title/msg/undo 버튼 구성...
  stack.prepend(el);

  let timer = setTimeout(() => dismiss(el), 4000);
  el.onmouseenter = () => clearTimeout(timer);          // hover 시 멈춤
  el.onmouseleave = () => timer = setTimeout(() => dismiss(el), 1500);

  if (undo) el.querySelector('.t-undo').onclick = () => { undo(); dismiss(el); };
}
function dismiss(el){ el.classList.add('leaving');
  el.addEventListener('animationend', () => el.remove(), {once:true}); }

♿ 접근성·상태 설계

  • aria-live로 낭독: 스택 컨테이너에 aria-live="polite"(중요·에러는 role="alert"=assertive)를 두면 새 토스트 텍스트가 자동으로 읽힙니다. 토스트마다 낱개로 붙이기보다 컨테이너에 한 번 두는 게 정석입니다.
  • 충분한 시간·hover 정지: 자동 소멸은 읽을 시간을 줘야 합니다(짧은 글 4초±). hover/focus 시 타이머를 멈춰 마우스·키보드 사용자가 놓치지 않게 합니다(WCAG의 "충분한 시간" 원칙).
  • 키보드 도달·닫기: 되돌리기·닫기는 진짜 <button>으로 Tab 도달·Enter 실행이 되게 합니다. 중요한 액션(되돌리기)을 자동 소멸 뒤로 숨기면 접근성상 위험하니 시간을 넉넉히 주세요.
paysable 적용: 정산 요청 전송·환불 처리·설정 저장 결과 알림에 씁니다. 특히 삭제/취소엔 되돌리기 토스트를 붙이면 실수 복구가 쉬워 지원 문의가 줍니다. 단 결제 승인/실패처럼 돈이 오가는 결정적 결과는 토스트로만 흘리지 말고 화면 본문·영수증에 확정 상태로 남기세요(토스트는 보조 알림).

🌀 CSS 앵커 포지셔닝 + Popover — JS 위치계산 라이브러리 없이 트렌드·개념

앵커 포지셔닝(CSS Anchor Positioning)은 어떤 요소를 다른 요소(앵커)에 붙여 위치시키는 CSS 기능입니다. 지금까진 툴팁·드롭다운·메뉴를 정확히 붙이려면 Floating UI(옛 Popper.js) 같은 JS 라이브러리로 좌표를 계산했는데, 이제 CSS 몇 줄로 됩니다. 여기에 Popover API(popover 속성)를 얹으면 열고 닫기·바깥 클릭·Esc 닫기·최상단 레이어까지 공짜입니다. 아래 툴팁·메뉴를 눌러 보고, 스크롤 상자 안에서 공간이 부족할 때 자동으로 위아래가 뒤집히는지(flip) 보세요.

툴팁 (anchor + popover)

결제 건당 2.9% + ₩100의 수수료가 정산 시 자동 차감됩니다.

버튼 위에 붙습니다. 위가 좁으면 position-try-fallbacks로 아래로 뒤집힙니다.

메뉴 (anchor + popover)

이 상자를 스크롤해 아래 버튼을 위쪽으로 밀어 보세요. 메뉴를 열면 공간이 없는 쪽을 피해 자동으로 방향이 바뀝니다.

확인 중…

열림/닫힘·바깥 클릭·Esc 닫기는 Popover API가, 붙는 위치와 뒤집기(flip)는 앵커 포지셔닝이 담당합니다 — 둘 다 JS 좌표 계산 없이.

📈 왜 지금 뜨나 / 어디에 쓰나

  • JS 위치 라이브러리를 대체: 툴팁·드롭다운·콤보박스·팝오버의 좌표 계산·뒤집기·화면 밖 방지 로직을 anchor()·position-anchor·position-try-fallbacks로 CSS화합니다. 번들에서 Floating UI를 걷어낼 수 있습니다(요구 브라우저가 받쳐줄 때).
  • Popover API와 궁합: popover 속성 + popovertarget 버튼이면 JS 없이 토글됩니다. top layer(최상단 레이어)에 떠서 overflow:hidden·z-index 지옥·부모 transform에 잘리는 고질병에서 벗어납니다.
  • 지원 현황(검색 기준, 2026-07): 앵커 포지셔닝은 Baseline 2026으로, Chrome·Edge 125+(2024), Safari 26(2025), Firefox 147(2026 초)에서 지원한다고 보고됩니다(자료마다 Firefox 버전 표기가 132~147로 갈려 정확한 최소 버전은 추정). Popover API는 Baseline 2024로 더 넓게 깔렸습니다. 아직 미지원 사용자를 위해 정적 위치 폴백을 두는 게 안전합니다.

💻 코드 (복붙용)

<!-- Popover API: JS 없이 토글 -->
<button popovertarget="tip1" aria-describedby="tip1">ⓘ 수수료 안내</button>
<div id="tip1" popover class="tip">건당 2.9% + ₩100</div>
/* 앵커 이름 붙이기 */
button[popovertarget="tip1"]{ anchor-name:--tipbtn; }

.tip{
  position:absolute;
  position-anchor:--tipbtn;      /* 이 앵커 기준으로 */
  bottom:anchor(top);           /* 앵커의 top 에 내 bottom 을 맞춤(=위에 붙음) */
  justify-self:anchor-center;   /* 가로 가운데 정렬 */
  margin-bottom:8px;
  position-try-fallbacks:flip-block; /* 위가 좁으면 아래로 자동 뒤집기 */
}

♿ 접근성·상태 설계

  • 관계를 ARIA로: 툴팁은 트리거에 aria-describedby, 메뉴 버튼은 aria-haspopup·(열림 시)aria-expanded를 줍니다. 위치는 CSS가 잡아도 의미 연결은 ARIA가 해야 낭독기가 압니다.
  • Popover의 기본 접근성: popover는 Esc 닫기·바깥 클릭 닫기·포커스 관리 일부를 브라우저가 처리합니다. 다만 메뉴 내부 화살표 이동 같은 세부 키보드 패턴은 직접 보완해야 합니다(순수 메뉴면 role="menu" 규약 고려).
  • 폴백 필수: 미지원 브라우저에선 앵커가 무시되어 요소가 엉뚱한 곳에 뜰 수 있으므로, @supports(anchor-name:--x)로 감지하거나 정적 position 기본값을 먼저 깔아 두세요.
paysable 적용: 거래 행의 ⋯ 액션 메뉴, 수수료·상태 툴팁, 날짜·카테고리 드롭다운에 씁니다. Next.js에선 이런 UI에 흔히 Floating UI/Radix를 쓰는데, 지원 브라우저가 확정되면 앵커+Popover로 번들을 줄이는 선택지가 생깁니다. 지금 단계에선 점진적 향상(폴백 유지)으로 실험하기 좋습니다.
