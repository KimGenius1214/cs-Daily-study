# HTTP/1.1 vs HTTP/2 vs HTTP/3 차이

> **Category**: 🌐 네트워크/브라우저 · **Level**: 기초 · **Date**: 2026-05-20 (수) · **실무연결도**: ⭐⭐⭐
> **Tags**: HTTP, HTTP/2, HTTP/3, QUIC, multiplexing

## 🎯 핵심 한 줄

같은 HTTP 시맨틱(메서드/헤더/상태코드)을 더 빠르고, 더 적게 막히는 전송 계층 위에 얹는 진화. 1.1 = 텍스트·순차, 2 = 바이너리·다중화, 3 = UDP·QUIC.

## 📖 개념 설명

HTTP는 "요청-응답"이라는 추상이 본질이고, 버전마다 바뀐 것은 **그 추상을 실어 나르는 방식**이다. 브라우저가 같은 origin으로 50개 요청을 보낼 때 어떻게 보내는지가 버전마다 완전히 다르다.

- **HTTP/1.1 (1997)**: 텍스트 프로토콜. 한 TCP 연결당 한 번에 한 요청만 in-flight. 그래서 브라우저는 origin당 보통 6개 connection을 병렬로 연다. Keep-Alive로 연결은 재활용하지만 응답이 느린 요청이 뒤 요청을 막는 **HoL blocking**이 application layer에서 발생.
- **HTTP/2 (2015)**: 한 TCP 연결 위에 여러 stream을 다중화(multiplexing). 모든 frame이 binary. HPACK으로 헤더 압축. server push도 있었지만 실패해서 deprecated. 다만 stream은 HTTP layer에서 다중화돼도, **TCP는 여전히 단일 byte stream**이라 한 패킷 손실이 전체 connection을 정지시키는 **TCP-level HoL blocking**이 남는다.
- **HTTP/3 (2022 표준화)**: TCP를 버리고 UDP 위에 **QUIC** 프로토콜을 새로 깖. QUIC이 stream을 OS 커널이 아니라 user space에서 독립적으로 처리하므로 한 stream 패킷 손실이 다른 stream을 막지 않는다. 0-RTT 재접속 지원으로 재방문 시 핸드셰이크 비용 절감.

## ⚙️ 동작 원리

### 한 origin에 60 KB짜리 응답 6개를 받는다고 가정

**HTTP/1.1**: 브라우저가 6 connection 동시에 열고 각자 하나씩 받음. TLS 핸드셰이크 6번. 7번째 요청은 뭐 하나 끝날 때까지 대기.

**HTTP/2**: TLS 1번 + TCP 1번 → 6 stream을 단일 connection에 인터리브. 헤더는 HPACK 정적/동적 테이블로 압축 (반복 헤더 1-2 바이트로 줄어듦). 단, 중간 라우터에서 패킷 하나 손실 시 TCP는 retransmit 받을 때까지 그 뒤 모든 데이터를 application에 못 올림 → 6 stream 다 멈춤.

**HTTP/3**: QUIC connection 1개 위에 6 stream. 한 stream의 손실은 그 stream만 멈추고 나머지 5개는 계속 흘러간다. 첫 연결은 1-RTT (TLS 1.3 통합), 재방문은 0-RTT로 GET 가능.

```
1.1:  [TCP1: req1───res1───] [TCP2: req2───res2───] [TCP3: req3───res3───]
2:    [TCP: TLS─[s1][s2][s3][s4][s5][s6]── ⛔ pkt loss → all stalled
3:    [UDP+QUIC: [s1][s2][s3][s4][s5][s6]── ⛔ s2 loss → only s2 stalled
```

## 💻 코드 예제

브라우저에서 현재 응답이 어떤 프로토콜로 왔는지 확인 (Resource Timing API):

```typescript
// utils/http-protocol.ts
type Protocol = 'http/1.1' | 'h2' | 'h3' | 'unknown';

/**
 * 특정 URL의 마지막 fetch 가 어떤 프로토콜로 처리됐는지 확인
 * Chrome/Firefox: `nextHopProtocol` 필드로 노출됨
 */
export function getProtocolForUrl(url: string): Protocol {
  const entry = performance
    .getEntriesByType('resource')
    .reverse()
    .find((e): e is PerformanceResourceTiming =>
      e.name === url && 'nextHopProtocol' in e
    );

  const raw = entry?.nextHopProtocol ?? '';
  if (raw === 'h3' || raw === 'h3-29') return 'h3';
  if (raw === 'h2') return 'h2';
  if (raw === 'http/1.1') return 'http/1.1';
  return 'unknown';
}

// 결제 API 응답이 어떤 프로토콜로 왔는지 모니터링
async function payRequest(orderId: string) {
  const url = '/api/payments/confirm';
  const res = await fetch(url, {
    method: 'POST',
    body: JSON.stringify({ orderId }),
  });

  // 다음 tick 까지 기다려야 Resource Timing 에 반영됨
  await new Promise((r) => setTimeout(r, 0));
  console.log('[payment] protocol =', getProtocolForUrl(url));

  return res.json();
}
```

DevTools Network 탭에서 'Protocol' 열을 켜면 `h2`/`h3` 가 한눈에 보인다. 위 코드는 그걸 런타임에 수집해 분석/알람 보내는 용도.

## 🏢 실무 사용 사례

- **paysable 같은 결제 도메인**: 결제 confirm → 영수증 GET → 토스트 표시 등 같은 origin에서 여러 호출이 짧게 연쇄 발생. HTTP/2 이상이면 단일 connection 재사용해 TLS 비용 없음. PG 콜백 endpoint도 HTTP/3 지원하면 모바일/Wi-Fi 전환 시 connection ID 유지로 끊김 적음.
- **이미지 많은 페이지**: Next.js `Image` 가 동일 origin (또는 CDN) 으로 수십 개 요청. HTTP/2 multiplexing 안 켜져 있으면 6개씩 큐잉돼서 LCP 망함.
- **GraphQL persistent connection**: HTTP/2 면 단일 connection 위에 batch 가능. HTTP/1.1 이면 connection 풀 고갈 위험.
- **국가 간 결제 트랜잭션**: latency 200ms+ 환경에서 HTTP/3 의 0-RTT 가 retry/조회 응답 속도에 체감 차이.
- **CDN 캐시 미스 fallback**: origin 까지 갈 때 HTTP/3 면 packet loss 가 많은 모바일 환경에서 stall 감소.

## ⚠️ 주의점 / 흔한 실수

- **HTTP/2 라고 무조건 빠르지 않다**: 도메인 샤딩 (`img1.example.com`, `img2.example.com` ...) 하던 코드를 그대로 두면 HTTP/2 의 single-connection 장점을 까먹는다. **샤딩 풀어서 한 origin 으로 모아야** 의미가 생긴다.
- **HTTP/3 는 클라이언트·서버·중간 인프라 모두 지원해야** 동작. 회사 방화벽이 UDP 443 막아두면 자동으로 HTTP/2 fallback.
- `Connection: close`, `Keep-Alive` 같은 hop-by-hop 헤더는 HTTP/2 에선 금지. 라이브러리 코드 남아있으면 RST_STREAM 받음.
- 서버 push 는 deprecated 상태 (Chrome 106+). 코드에 남아있으면 정리 권장. 대신 `103 Early Hints` 활용.
- Long-poll / WebSocket 은 HTTP/2 multiplexing 으로도 connection 점유. SSE 가 더 친화적.
- 측정은 반드시 **production-like 환경**에서. 로컬은 RTT 0 이라 HTTP/3 이점이 안 보인다.
