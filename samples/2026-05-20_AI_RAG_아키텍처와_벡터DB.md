# RAG 아키텍처와 벡터 DB

> **Category**: 🤖 AI/ML/LLM · **Level**: 응용 · **Date**: 2026-05-20 (수) · **실무연결도**: ⭐⭐
> **Tags**: RAG, embedding, vector DB, semantic search, LLM

## 🎯 핵심 한 줄

LLM 의 지식 한계를 외부 문서로 보완하기 위해 **"질문 → 의미가 가까운 문서 검색 → 검색 결과를 프롬프트에 끼워 호출"** 로 답을 만드는 패턴. 벡터 DB 는 그 "의미적으로 가까운"을 빠르게 찾기 위한 인프라.

## 📖 개념 설명

LLM 만 가지고 답하려고 하면 두 가지가 막힌다.
- **지식 cutoff**: 모델이 학습된 시점 이후 데이터 모름.
- **사내 데이터 부재**: paysable 매뉴얼, 약관, 내부 위키 같은 건 학습된 적 없음.

이 두 문제를 모델을 재학습(fine-tuning) 없이 해결하려는 가장 보편적인 패턴이 **RAG (Retrieval-Augmented Generation)** 다. 구조는 두 개 phase 로 단순화된다.

- **Ingestion phase** (오프라인, 1회 또는 주기적):
  1. 문서를 의미 단위로 잘라 (chunking)
  2. 각 chunk 를 임베딩 모델로 벡터화하고
  3. 벡터 DB 에 (chunk, 벡터, metadata) 저장
- **Query phase** (런타임, 사용자 요청마다):
  1. 질문도 같은 임베딩 모델로 벡터화
  2. 벡터 DB 에서 cosine similarity 기준 top-k chunk 검색
  3. 검색된 chunk 를 system prompt 에 끼워 LLM 호출
  4. 결과를 사용자에게 반환

벡터 DB 는 단순히 cosine similarity 를 계산해주는 게 본질이 아니라, **수백만~수억 벡터 중 가장 가까운 k 개를 sub-linear time 에 찾기 위한 ANN (Approximate Nearest Neighbor) 인덱스**를 제공하는 게 핵심이다. HNSW, IVF 같은 알고리즘이 그 안에서 돈다.

## ⚙️ 동작 원리

### 임베딩이란

텍스트를 1024 또는 1536 차원 같은 고차원 벡터로 변환한 것. "의미가 비슷한 단어/문장은 벡터 공간에서 가까이 위치한다"는 성질을 가지도록 학습된 결과물. cosine similarity (두 벡터의 사잇각 코사인) 가 높을수록 의미적으로 가깝다고 본다.

### 청킹 전략

문서를 한 덩어리로 임베딩하면 retrieval 정확도가 떨어진다.
- **Fixed-size**: 500 토큰 단위로 자르고 50 토큰 overlap. 가장 단순.
- **Semantic chunking**: 문단/섹션 기반으로 자르고 너무 크면 분할. 정확도 ↑.
- **Recursive**: 큰 단위(문서→섹션→문단→문장) 로 fallback 잘라가며 토큰 한도 맞춤.
- **Sliding window with overlap**: 경계에서 의미 끊김 방지.

청킹은 검색 품질의 60% 를 좌우한다고 봐도 무방.

### 벡터 DB 선택지

| 옵션 | 특징 | 적합 케이스 |
|---|---|---|
| **pgvector** (Postgres extension) | 기존 RDB 트랜잭션과 한 DB | 데이터 수십만, 권한 모델 RDB 와 통합 |
| **Pinecone** | SaaS, serverless | 빠른 시작, 운영 부담 X |
| **Weaviate / Qdrant** | self-host 가능 | 온프레 / 데이터 주권 |
| **Chroma** | local-first, Python 친화 | 개발/PoC |
| **OpenSearch / Elasticsearch** | full-text + vector 하이브리드 | 키워드 검색 병행 필요 |

### Retrieval 만으로는 부족 — Re-ranking

top-k 로 뽑힌 chunk 가 진짜 답을 가진 chunk 가 아닐 때가 많다. 그래서 1차로 50개 정도 후보를 뽑고 cross-encoder(예: cohere rerank, bge-reranker)로 다시 점수 매겨 상위 5개만 LLM 에 넣는 패턴이 표준이 돼간다.

## 💻 코드 예제

Next.js API Route 에서 pgvector 기반 RAG. paysable FAQ 챗봇이라고 가정.

```typescript
// app/api/chat/route.ts
import { OpenAI } from 'openai';
import { sql } from '@vercel/postgres';
import { type NextRequest } from 'next/server';

const openai = new OpenAI();
const EMBED_MODEL = 'text-embedding-3-small'; // 1536 dim
const CHAT_MODEL = 'gpt-4o-mini';

interface Chunk {
  id: number;
  source: string;
  content: string;
  similarity: number;
}

async function embed(text: string): Promise<number[]> {
  const res = await openai.embeddings.create({
    model: EMBED_MODEL,
    input: text,
  });
  return res.data[0].embedding;
}

async function retrieveTopK(query: string, k = 5): Promise<Chunk[]> {
  const queryVec = await embed(query);
  // pgvector: <=> = cosine distance. 작을수록 가까움.
  const { rows } = await sql<Chunk>`
    SELECT id, source, content, 1 - (embedding <=> ${JSON.stringify(queryVec)}::vector) AS similarity
    FROM faq_chunks
    ORDER BY embedding <=> ${JSON.stringify(queryVec)}::vector
    LIMIT ${k}
  `;
  return rows.filter((r) => r.similarity > 0.5);
}

export async function POST(req: NextRequest) {
  const { question } = await req.json();
  const chunks = await retrieveTopK(question, 5);

  const context = chunks
    .map((c, i) => `[${i + 1}] (${c.source})\n${c.content}`)
    .join('\n\n---\n\n');

  const stream = await openai.chat.completions.create({
    model: CHAT_MODEL,
    stream: true,
    messages: [
      {
        role: 'system',
        content:
          '너는 paysable 결제 서비스의 고객 지원 봇이다. 아래 컨텍스트만 근거로 답하고, ' +
          '컨텍스트에 없는 내용은 "확인이 필요합니다"라고 답한다. ' +
          '인용 시 [번호] 형식으로 출처를 표기하라.\n\n' +
          `=== 컨텍스트 ===\n${context}`,
      },
      { role: 'user', content: question },
    ],
  });

  // SSE 로 프론트에 흘림 (Vercel AI SDK 쓰면 더 간결)
  const encoder = new TextEncoder();
  const readable = new ReadableStream({
    async start(ctrl) {
      for await (const chunk of stream) {
        const delta = chunk.choices[0]?.delta?.content ?? '';
        if (delta) ctrl.enqueue(encoder.encode(`data: ${JSON.stringify({ delta })}\n\n`));
      }
      ctrl.close();
    },
  });

  return new Response(readable, {
    headers: { 'Content-Type': 'text/event-stream' },
  });
}
```

핵심은 `retrieveTopK` 한 번 + LLM 호출 한 번. **답이 잘 안 나오면 99% retrieval 단계가 문제**고, prompt 튜닝보다 청킹/임베딩 모델 교체가 효과 크다.

## 🏢 실무 사용 사례

- **결제 FAQ 챗봇**: 약관/매뉴얼/CS 케이스를 ingest 해두고 사용자 질문에 인용 답변. PG 별 차이 같은 디테일에 강함.
- **개발자 문서 검색**: paysable SDK 문서를 ingest. "구독 결제 취소 어떻게?" 같은 자연어 질문으로 코드 스니펫 검색.
- **사내 위키 Q&A**: Notion/Confluence 정기 sync → 임베딩 → 슬랙 봇.
- **로그/이슈 검색**: Sentry 에러 메시지 임베딩해 "비슷한 에러 과거 발생한 적 있나?" 검색.
- **추천**: 사용자 클릭한 상품 임베딩과 유사한 상품 top-k 로 "유사 상품" 표시. (벡터 DB 의 가장 오래된 용도 중 하나)

## ⚠️ 주의점 / 흔한 실수

- **임베딩 모델은 ingest 와 query 가 반드시 동일해야 한다**. 모델 바꾸려면 전체 reindex.
- **청크가 너무 크면** LLM context 토큰 폭주, **너무 작으면** 문맥이 짤려 답이 안 나옴. 256~512 토큰이 보통.
- **cosine distance 임계값(예: > 0.5) 안 두면** 무관한 chunk 가 prompt 에 들어가 환각 ↑.
- **metadata filtering** 잊지 말기. 부서/언어/시점별로 filter 안 하면 retrieval 정확도 폭락. 예: WHERE source IN ('manual', 'faq') AND lang='ko'.
- **PII / 카드번호** ingest 전에 마스킹. RAG 결과가 그대로 LLM provider 에 전송됨.
- **단순 RAG 가 만능 아님**. 표 계산, 다중 hop 추론, 정확한 숫자 비교는 RAG 보다 SQL agent / tool use 가 낫다.
- **재현성**: 같은 질문에 같은 답을 원한다면 retrieval 결과를 로그로 남겨야 디버깅 가능.

## 🤔 심화 질문 3개

1. **(면접 빈출)** 벡터 DB 가 cosine similarity 를 계산해주는 데 왜 굳이 ANN 인덱스가 필요한가? Brute force 와 비교해 어느 규모부터 의미가 있나?
2. **(응용/연결)** retrieval top-k 의 recall 을 측정하고 싶다. 평가셋을 어떻게 만들고 어떤 메트릭(Recall@k, MRR, NDCG)을 보겠는가?
3. **(트러블슈팅)** 사용자가 "환불 정책 알려줘" 라고 했는데 봇이 "결제 한도" 답을 한다. retrieval 까지는 정상이라 가정하고, 무엇부터 점검하겠는가? (힌트: 임베딩 모델의 의미적 거리, chunk 헤더 정보, prompt 안 컨텍스트 순서)
